**重要な事実**:
- ❌ ミニアプリ内QRスキャン: 削除後も「取得済み」エラー
- ✅ カメラQRスキャン: 削除後に正常にスキャン可能

## 🔍 調査結果

### コードレベルの検証

1. ✅ **削除APIとスキャンAPIは同じタイムゾーン計算を使用**
   - 両方とも `new Date().getTime() + 9 * 60 * 60 * 1000` で JST に変換
   - 両方とも `${todayJST}T00:00:00+09:00` ~ `${todayJST}T23:59:59.999+09:00`

2. ✅ **両方とも同じSupabaseクライアントを使用**
   - 削除API: `import { supabase } from "@/lib/supabase"`
   - スキャンAPI: `import { supabase } from "@/lib/supabase"`
   - 両方とも ANONキー + RLS有効

3. ✅ **同じ検索条件**
   - `.eq("user_id", userId)`
   - `.eq("stamp_method", "qr")`
   - `.gte("visit_date", startOfDay)` + `.lt("visit_date", endOfDay)`

4. ✅ **RLSポリシーとトリガーは正しく実装済み**
   - `supabase/016B_add_delete_policy_stamp_history.sql` で DELETE を許可
   - DELETE トリガーで `profiles.stamp_count` を再計算

### カメラQRスキャンは成功する理由

カメラQRスキャン (`/api/stamps/auto`) は削除後に正常動作する
→ **削除APIは正しく動作している**
→ **Supabaseのデータは正しく削除されている**

## 🔬 考えられる原因

### 仮説1: 削除APIが実際には呼ばれていない

**検証方法**:
- Vercelログで「🗑️ [Delete Today QR]」が出力されているか確認
- もし出力されていない → フロントエンドで削除ボタンが正しく動作していない

**対策**:
- [components/shared/StaffPinModal.tsx](../components/shared/StaffPinModal.tsx) の 95-128行目を確認
- `handleDeleteTodayQR` 関数が呼ばれているか

### 仮説2: 削除APIが部分的にしか削除していない

**検証方法**:
- Vercelログで「削除クエリ結果」の `count` を確認
- `count: 0` の場合 → 削除対象が見つかっていない
- `count: 1` だが再スキャン時に `recordCount: 1` → 別のレコードが残っている

**対策**:
- 削除API実行後の「🔍 [Delete Today QR] 削除後の本日QRレコード数」を確認
- 0件以外なら、削除条件が間違っている

### 仮説3: タイムゾーンの微妙なズレ

**検証方法**:
- 削除API実行時の `startOfDay` / `endOfDay`
- スキャンAPI実行時の `startOfDayUTC` / `endOfDayUTC`
- 両者が完全に一致しているか確認

**対策**:
- Vercelログの `visit_date` と検索範囲を比較
- 範囲外なら、タイムゾーン計算を修正

### 仮説4: Supabaseキャッシュ（可能性低い）

**検証方法**:
- 削除直後に数秒待ってから再スキャン
- それでも問題が起きるか

**対策**:
- Supabaseクエリに `{ cache: 'no-cache' }` オプションを追加

### 仮説5: 複数のデバイス/セッション

**検証方法**:
- 削除したユーザーIDと再スキャンしたユーザーIDが一致しているか
- Vercelログの `userId` を比較

**対策**:
- 本人確認を強化
