# バグ報告: スタンプ減少時の特典削除未対応

**発見日:** 2026-04-04
**重要度:** 🔴 **高** (データ不整合)
**ステータス:** ✅ **修正完了**
**修正日:** 2026-04-04

---

## 📋 バグの概要

スタッフがスタンプ数を手動で減らした場合（例: 200個 → 0個）、**スタンプ数は更新されるが、既に付与された特典が削除されない**。

### 具体例

| 操作 | profiles.stamp_count | reward_exchanges | milestone_history |
|------|---------------------|------------------|-------------------|
| **初期状態** | 200個 | 20件（10,20,30...200） | 20件 |
| **スタッフ操作: 0個に変更** | ✅ 0個 | ❌ 20件残る | ❌ 20件残る |
| **期待される状態** | ✅ 0個 | ✅ 0件 | ✅ 0件 |

**結果:** ユーザーはスタンプ0個なのに、200個分の特典を保持し続ける ❌

---

## 🔍 根本原因

### 1. マイルストーン判定ロジックの制限

[lib/milestones.ts](../lib/milestones.ts) の `checkMilestones()` 関数:

```typescript
export function checkMilestones(
  oldStampCount: number,
  newStampCount: number
): MilestoneResult[] {
  const results: MilestoneResult[] = [];

  // 通過した全マイルストーンを取得
  const allMilestones: number[] = [];

  // 10の倍数を収集
  const old10 = Math.floor(oldStampCount / 10);
  const new10 = Math.floor(newStampCount / 10);
  for (let i = old10 + 1; i <= new10; i++) {  // ← ここが問題！
    allMilestones.push(i * 10);
  }
  // ...
}
```

**問題点:**
- `for (let i = old10 + 1; i <= new10; i++)` は**増加時のみ動作**
- `oldStampCount > newStampCount` の場合、ループが実行されない
- 結果: `results` が空配列 `[]` で返る

### 2. スタッフ手動変更APIの未対応

[app/api/stamps/manual/route.ts](../app/api/stamps/manual/route.ts):

```typescript
export async function POST(request: NextRequest) {
  // ... (省略)

  // スタンプ数を直接更新（99-153行目）
  await supabase.from("stamp_history").insert({ ... });
  await supabase.from("profiles").update({ stamp_count: newStampCount });

  // ⚠️ マイルストーン判定なし
  // ⚠️ 特典削除処理なし

  return NextResponse.json({ success: true, stampCount: newStampCount });
}
```

**問題点:**
- マイルストーン判定を**全く呼んでいない**
- スタンプが減った場合の特典削除ロジックが**存在しない**

## 🛠️ 修正方針

### オプション A: ソフトデリート（推奨）

**メリット:**
- データの監査証跡が残る
- 誤操作時に復元可能
- 既存のRLS/制約に影響なし

**実装:**
```typescript
// lib/milestones.ts に追加
export async function invalidateMilestoneRewards(
  userId: string,
  oldStampCount: number,
  newStampCount: number
) {
  if (newStampCount >= oldStampCount) return; // 増加時は何もしない

  // 削除すべきマイルストーンを計算
  const old10 = Math.floor(oldStampCount / 10);
  const new10 = Math.floor(newStampCount / 10);

  const milestonearray = [];
  for (let i = new10 + 1; i <= old10; i++) {
    milestonesToRemove.push(i * 10);
  }

  if (milestoneesToRemove.length === 0) return;

  // reward_exchanges をソフトデリート（status = 'cancelled'）
  const { error: rewardError } = await supabase
    .from('reward_exchanges')
    .update({
      status: 'cancelled',
      notes: `スタッフ操作により無効化 (${oldStampCount} → ${newStampCount})`
    })
    .eq('user_id', userId)
    .eq('is_milestone_based', true)
    .in('milestone_reached', milestoneesToRemove)
    .in('status', ['pending', 'approved']); // 既に使用済み('used')は無効化しない

  // milestone_history も記録（削除ではなく無効化イベントとして）
  // オプション: 削除する場合は DELETE、記録する場合は何もしない
}
```

**適用箇所:**
```typescript
// app/api/stamps/manual/route.ts の 153行目あたりに追加
const changeAmount = newStampCount - currentStampCount;

// マイルストーン特典の無効化処理（スタンプ減少時）
if (changeAmount < 0) {
  try {
    await invalidateMilestoneRewards(userId, currentStampCount, newStampCount);
    console.log(`✅ マイルストーン特典を無効化: User ${userId}, ${currentStampCount} → ${newStampCount}`);
  } catch (error) {
    console.error(`❌ マイルストーン特典無効化エラー:`, error);
    // エラーでもスタンプ数変更は完了しているので処理続行
  }
}
```

### オプション B: 物理削除（非推奨）

**メリット:**
- データが完全にクリーンになる

**デメリット:**
- 監査証跡が失われる
- 誤操作時に復元不可能
- 'used' 状態の特典も削除してしまう可能性

**実装:**
```typescript
// reward_exchanges を物理削除
await supabase
  .from('reward_exchanges')
  .delete()
  .eq('user_id', userId)
  .eq('is_milestone_based', true)
  .in('milestone_reached', milestoneesToRemove)
  .in('status', ['pending', 'approved']); // 使用済みは削除しない

// milestone_history も物理削除
await supabase
  .from('milestone_history')
  .delete()
  .eq('user_id', userId)
  .in('milestone', milestoneesToRemove);
```

**懸念点:**
- `milestone_history` には `reward_exchange_id` の外部キー制約がある可能性
- 削除順序を誤るとエラー

## ✅ 修正内容

### 実装されたコード

#### 1. [lib/milestones.ts](../lib/milestones.ts) (266-325行目)

```typescript
export async function invalidateMilestoneRewards(
  userId: string,
  oldStampCount: number,
  newStampCount: number
): Promise<number> {
  // 増加時は何もしない
  if (newStampCount >= oldStampCount) return 0;

  // 削除すべきマイルストーンを計算
  const old10 = Math.floor(oldStampCount / 10);
  const new10 = Math.floor(newStampCount / 10);
  const milestoneArray: number[] = [];

  for (let i = new10 + 1; i <= old10; i++) {
    milestoneArray.push(i * 10);
  }

  if (milestoneArray.length === 0) return 0;

  // reward_exchanges をソフトデリート（status = 'cancelled'）
  const { data, error } = await supabase
    .from('reward_exchanges')
    .update({
      status: 'cancelled',
      notes: `スタッフ操作により無効化 (${oldStampCount} → ${newStampCount})`,
      updated_at: new Date().toISOString()
    })
    .eq('user_id', userId)
    .eq('is_milestone_based', true)
    .in('milestone_reached', milestoneArray)
    .in('status', ['pending', 'approved']); // 使用済み('used')は除外

  return data?.length || 0;
}
```

#### 2. [app/api/stamps/manual/route.ts](../app/api/stamps/manual/route.ts) (158-181行目)

```typescript
// マイルストーン特典の無効化処理（スタンプ減少時のみ）

  try {
    const invalidatedCount = await invalidateMilestoneRewards(
      userId,

    console.error(`❌ マイルストーン特典無効化エラー:`, error);
```

---

### 実装の特徴

1. **ソフトデリート採用**
   - `status` を `'cancelled'` に変更（物理削除しない）
   - 監査証跡として記録が残る
   - 誤操作時に復元可能

2. **使用済み特典を保護**
   - `.in('status', ['pending', 'approved'])` で `'used'` を除外
   - 既に使われた特典は無効化しない

3. **milestone_history は削除しない**
   - 「過去にこのマイルストーンに到達した」記録を保持
   - `reward_exchanges.status='cancelled'` で無効化状態を判別可能

4. **エラーハンドリング**
   - 無効化処理でエラーが発生してもスタンプ数変更は成功扱い
   - ログに警告を出力し、管理ダッシュボードで手動確認を促す

### 既存不整合データへの対応

**方針:** 管理ダッシュボード側で手動削除（ユーザー確認済み）

今回の修正は**今後のスタッフ操作のみ**に適用されます。既に不整合が発生しているユーザーについては、以下のSQLで検出可能です:

```sql
-- 不整合ユーザーの検出
SELECT
  p.id AS user_id,
  p.display_name,
  p.stamp_count,
  COUNT(r.id) AS invalid_reward_count,
  ARRAY_AGG(r.milestone_reached ORDER BY r.milestone_reached) AS invalid_milestones
FROM profiles p
JOIN reward_exchanges r ON (
  r.user_id = p.id
  AND r.is_milestone_based = true
  AND r.status IN ('pending', 'approved')
  AND r.milestone_reached > FLOOR(p.stamp_count / 10) * 10
)
GROUP BY p.id, p.display_name, p.stamp_count
ORDER BY p.stamp_count ASC;
```

## 🔴 追加バグ発見: スタンプ復活問題（同日発見・修正）

**発見日:** 2026-04-04
**問題:** スタッフ操作で200個→0個後、QRスキャン(+15個)で**200個に復活**

### 根本原因

```typescript
// app/api/stamps/manual/route.ts (100-101行目のコメント)
// 注: トリガーは MAX(stamp_number) を使うため、減らす操作には対応できない
// そのため、手動変更では stamp_history への記録後に profiles.stamp_count を直接更新する
```

**動作フロー:**
1. スタッフ: 200→0 → `stamp_history`: `[200(qr), 0(manual_admin)]`
2. `profiles` を直接UPDATE → `stamp_count = 0` ✅
3. QRスキャン: +15 → `stamp_history`: `[200, 0, 15]`
4. トリガー: `stamp_count = MAX(200, 0, 15) = 200` ❌

**トリガーSQL:**
```sql
-- supabase/002_create_stamp_history_table.sql
stamp_count = (
  SELECT COALESCE(MAX(stamp_number), 0)
  FROM stamp_history
  WHERE user_id = NEW.user_id  -- 全履歴から最大値を取得
)
```

### 修正方針

**採用: 古い履歴を削除（最もシンプル）**

- スタッフ操作 = 「新しい起点」として扱う
- 古い `stamp_history` レコードを全削除
- `manual_admin` レコードのみ残す（監査証跡）
- トリガーSQL変更不要
- ミニアプリ側で完結

**修正コード:**

```typescript
// app/api/stamps/manual/route.ts (113-133行目)

// スタッフ操作を「起点」として扱う: 古い履歴を削除
const { error: deleteError } = await supabase
  .from("stamp_history")
  .delete()
  .eq("user_id", userId);

if (deleteError) {
  console.error("❌ 古い履歴削除エラー:", deleteError);
  return NextResponse.json({
    success: false,
    message: "古い履歴の削除に失敗しました",
    error: deleteError.message,
  }, { status: 500 });
}

console.log(`🗑️ スタッフ操作を起点として設定: User ${userId} の古い履歴を削除`);

// その後、新しい起点として manual_admin レコードを INSERT
await supabase.from("stamp_history").insert({
  stamp_number: newStampCount,
  stamp_method: "manual_admin",
  amount: changeAmount,
  ...
});
```

**修正後の動作:**

| 操作 | stamp_history | profiles.stamp_count | トリガー計算 |
|------|--------------|---------------------|-------------|
| 初期状態 | [200(qr)] | 200 | - |
| スタッフ: 0個 | **[0(manual_admin)]** | 0 | MAX(0) = 0 ✅ |
| QRスキャン: +15 | [0, 15] | 15 ✅ | MAX(0, 15) = 15 ✅ |

**メリット:**
- トリガーSQL変更不要 ✅
- ミニアプリ側で完結 ✅
- シンプルで理解しやすい ✅
- `manual_admin` レコードが監査証跡として残る ✅

**デメリット:**
- 過去の `stamp_history` が失われる（管理ダッシュボードで履歴表示不可）
- ただし、スタッフ操作自体が「リセット」の意味なので妥当
