## 1. 対応方針サマリー

### 1.1 緊急度別分類

| 優先度 | 項目 | 対応方針 | 担当 |
|--------|------|---------|------|
| 🔴 **Critical** | カラム名の不一致（`stamp_count`, `display_name`） | **85/86を修正**（現状スキーマに合わせる） | LIFF担当 |
| 🔴 **Critical** | RLS・認証方式の不整合 | **方針を確定**（auth.uid()を使わない） | 要協議 |
| 🟡 **High** | `is_active`, `age` の存在しないカラム | **使わない方向で修正** | LIFF担当 |
| 🟡 **High** | 最終来院日の条件（`created_at` vs `visit_date`） | **`last_visit_date` / `visit_date` に統一** | LIFF担当 |
| 🟢 **Medium** | `postponed_count` の更新方法 | **正しい実装に修正** | LIFF担当 |
| 🟢 **Medium** | ダッシュボードのディレクトリ構成 | **`app/admin/`に読み替え** | ダッシュボード担当 |
| 🔵 **Low** | LINE配信と既存一斉配信の統合 | **Phase 2以降で検討** | 両者 |

### 1.2 基本方針

✅ **現状のスキーマ（03_Database_Schema.md）を優先**
- 85/86の仕様書を現状に合わせて修正
- 新規テーブル追加は最小限に

✅ **認証方式は現状の simple-auth + RLS緩和 を維持**
- `auth.uid()` 依存のRLSは使わない
- アプリ層で user_id による絞り込みを担保

✅ **段階的実装を推奨**
- Phase 1: 固定質問（q1, q2, q3）+ 基本配信機能
- Phase 2: JSONB動的質問 + 高度なフィルタ
- Phase 3: 管理画面UI + LINE自動配信

## 2. データベース・スキーマ関連

### 2.1 ❌ 存在しないカラムへの対応

#### **問題1: `profiles.is_active`**

**86での使用:**
```sql
-- 全員配信
select id from profiles where is_active = true;
```

**現状:** `is_active` カラムは存在しない

**対応方針:** ✅ **使わない**
- 全員配信時は `where` 条件なしで全件取得
- 将来的に「退会ユーザー」を管理したい場合のみ追加検討

**修正後のSQL:**
```sql
-- 全員配信（is_active 条件を削除）
select id from profiles;

-- または、スタンプ数0のユーザーを除外（実質的なアクティブ判定）
select id from profiles where stamp_count > 0;
```

---

#### **問題2: `profiles.age`**

**86での使用:**
```sql
-- 年齢フィルタ
where age >= 30;
```

**現状:** `age` カラムも `birth_date` カラムも存在しない

**対応方針:** ✅ **Phase 1では使わない**
- 年齢フィルタ機能は **Phase 2以降** で検討
- 必要になったら `birth_date DATE` カラムを追加し、年齢はSQL側で計算
  ```sql
  -- 将来実装する場合
  where EXTRACT(YEAR FROM AGE(birth_date)) >= 30;
  ```

**当面の対処:**
- 配信対象者フィルタから「年齢層」は除外
- 「最終来院日」「スタンプ数」のみで運用開始

---

#### **問題3: `profiles.stamps_count` → `stamp_count`**

**86での使用:**
```sql
where stamps_count >= 50;  -- ❌ 誤り
```

**現状:** 正しいカラム名は **`stamp_count`**（複数形ではない）

**対応方針:** ✅ **85/86のすべてのSQL例を `stamp_count` に修正**

**修正例:**
```sql
-- スタンプ5個以上（50pt = 5個）
where stamp_count >= 50;
```

---

#### **問題4: `profiles.line_display_name` → `display_name`**

**86での使用:**
```sql
-- ビュー定義
p.line_display_name
```

**現状:** 正しいカラム名は **`display_name`**

**対応方針:** ✅ **`display_name` に統一**

### 2.2 ❌ 日付カラムの取り違え

#### **問題: `stamp_history.created_at` を最終来院日として使用**

**86での使用（9.3 休眠ユーザー抽出）:**
```sql
having max(sh.created_at) < now() - interval '90 days'  -- ❌ 誤り
```

**現状の正しいカラム:**
- **`profiles.last_visit_date`**: 最終来院日（トリガーで自動更新される）
- **`stamp_history.visit_date`**: 実際の来院日
- **`stamp_history.created_at`**: レコード作成日時（≠来院日）

**対応方針:** ✅ **`profiles.last_visit_date` または `stamp_history.visit_date` を使用**

**修正後のSQL:**
```sql
-- 方法A: profiles のみで抽出（推奨：高速）
select id from profiles
where last_visit_date < now() - interval '90 days'
   or last_visit_date is null;

-- 方法B: stamp_history と結合（不要にJOINしない）
select p.id from profiles p
left join stamp_history sh on p.id = sh.user_id
group by p.id
having max(sh.visit_date) < now() - interval '90 days'
   or max(sh.visit_date) is null;
```

## 3. 認証・RLS関連

### 3.1 現状の認証方式の確認

| 環境 | 認証方式 | Supabaseクライアント | `auth.uid()` |
|------|---------|---------------------|-------------|
| **LIFF（患者側）** | LINEログインのみ（LIFF SDK） | anon key | ❌ NULL |
| **ダッシュボード（管理側）** | Cookie（simple-auth） | service_role | ❌ NULL |

**重要:**
- LIFF側は **Supabase Auth を使用していません**
- ユーザー識別は LIFF SDK の `liff.getProfile()` で取得した LINE user id を元に、`profiles` テーブルから `id` を取得
- したがって `auth.uid()` は**常に NULL** で使用不可

**結論:** `auth.uid()` は**使えない**

### 3.2 RLS方針の決定

#### ❌ 86の当初案（auth.uid()依存）

```sql
-- これは動かない
create policy "Users can view their own targets"
  on survey_targets for select
  using (auth.uid()::text = user_id);
```

#### ✅ 修正案A: RLSを緩和（アプリ層で制御）

```sql
-- LIFF（患者側）: anon key で自分の行のみ参照
create policy "anon_can_read_survey_targets"
  on survey_targets for select
  using (true);  -- 全件参照可能だが、アプリ側で user_id でフィルタ

create policy "anon_can_update_survey_targets"
  on survey_targets for update
  using (true);  -- 全件更新可能だが、アプリ側で user_id でフィルタ

-- ダッシュボード（管理側）: service_role で全件操作可能
-- ※ service_role は RLS をバイパスするため、ポリシー不要
```

**セキュリティ担保方法:**
- LIFF側: クエリに必ず `.eq('user_id', profile.id)` を付ける
- ダッシュボード側: service_role で直接操作

#### ✅ 修正案B: カスタムヘッダーで認証（より厳密）

```sql
-- アプリ側で X-User-ID ヘッダーを送信し、RLSで検証
create policy "authenticated_user_can_read"
  on survey_targets for select
  using (current_setting('request.headers')::json->>'x-user-id' = user_id);
```

**推奨:** ⭐ **修正案A（RLS緩和 + アプリ層制御）**
- 現状の認証方式と整合性が高い
- 実装が容易
- セキュリティはアプリ層で担保（既存のパターンと同じ）

## 4. 実装コード関連

### 4.1 「あとで」ボタンの実装修正

#### ❌ 86の当初案

```typescript
// これは動かない
postponed_count: supabase.rpc('increment', { row: 'postponed_count' })
```

#### ✅ 修正案A: SELECT → UPDATE

```typescript
const handleSurveyLater = async () => {
  // 1. 現在値を取得
  const { data: target } = await supabase
    .from('survey_targets')
    .select('postponed_count')
    .eq('user_id', userId)
    .eq('survey_id', surveyId)
    .single();

  // 2. +1 して更新
  await supabase
    .from('survey_targets')
    .update({
      postponed_count: (target?.postponed_count || 0) + 1,
      last_postponed_at: new Date().toISOString(),
    })
    .eq('user_id', userId)
    .eq('survey_id', surveyId);

  setSurveyModal(null);
};
```

#### ✅ 修正案B: RPC関数を作成（推奨）

**マイグレーション:**
```sql
-- 017_create_survey_tables.sql に追加
create or replace function increment_survey_postponed(
  p_user_id text,
  p_survey_id text
)
returns void as $$
begin
  update survey_targets
  set
    postponed_count = postponed_count + 1,
    last_postponed_at = now()
  where user_id = p_user_id
    and survey_id = p_survey_id;
end;
$$ language plpgsql security definer;
```

**LIFF側:**
```typescript
const handleSurveyLater = async () => {
  await supabase.rpc('increment_survey_postponed', {
    p_user_id: userId,
    p_survey_id: surveyId,
  });

  setSurveyModal(null);
};
```

**推奨:** ⭐ **修正案B（RPC関数）**
- トランザクション安全
- コードがシンプル
- 既存の `increment_reservation_clicks` と同じパターン

## 5. その他の調整

### 5.1 スタンプ報酬の付与方法

#### 要件
- アンケート回答で「スタンプ0.3個（3pt）」を付与
- 既存の積み上げ式と整合を取る

#### ✅ 実装方針

**API実装例（`app/api/survey/submit/route.ts`）:**
```typescript
// 1. 回答をsurvey_answersに保存
await supabase.from('survey_answers').insert({
  user_id: profile.id,
  survey_id: surveyId,
  q1_rating,
  q2_comment,
  q3_recommend,
});

// 2. スタンプ履歴に追加（既存システムと同じ）
await supabase.from('stamp_history').insert({
  user_id: profile.id,
  visit_date: new Date().toISOString(),
  stamp_number: profile.stamp_count + 3,  // 現在値 + 3pt
  amount: 3,                               // 今回付与: 3pt
  stamp_method: 'survey_reward',           // ★ 新規メソッド
  notes: `アンケート回答: ${surveyId}`,
});

// 3. トリガーで profiles.stamp_count が自動更新される

// 4. survey_targets の answered_at を更新
await supabase
  .from('survey_targets')
  .update({ answered_at: new Date().toISOString() })
  .eq('user_id', profile.id)
  .eq('survey_id', surveyId);
```

**ポイント:**
- `stamp_method` に新しい値 `'survey_reward'` を使用
- 既存のトリガー（`update_profile_stamp_count`）が自動的に `profiles.stamp_count` を更新
- 積み上げ式なので減算不要

### 5.3 配信対象者抽出の最適化

#### ❌ 86の例（不要なJOIN）

```sql
-- 最終来院日30日以内（stamp_history と JOIN している）
insert into survey_targets (user_id, survey_id, show_on_liff_open)
select p.id, 'satisfaction_2026Q1', true
from profiles p
join stamp_history sh on p.id = sh.user_id
where sh.created_at >= now() - interval '30 days'
group by p.id;
```

#### ✅ 修正案（profiles のみで抽出）

```sql
-- profiles だけで可能（高速）
insert into survey_targets (user_id, survey_id, show_on_liff_open)
select id, 'satisfaction_2026Q1', true
from profiles
where last_visit_date >= now() - interval '30 days';
```

**原則:**
- **profiles に集約されている情報（last_visit_date, stamp_count など）は profiles のみで抽出**
- stamp_history との JOIN は最小限に（パフォーマンス向上）

## 6. 修正版データベース設計

### 6.1 新規マイグレーション: `017_create_survey_tables.sql`

```sql
-- ========================================
-- アンケート機能テーブル作成
-- Phase 2.8: 患者満足度調査・マーケティング分析用
-- ========================================

-- ========================================
-- 1. surveys テーブル（アンケート定義）
-- ========================================
create table surveys (
  id text primary key,                      -- 'satisfaction_2026Q1' など
  title text not null,                      -- 'ご利用満足度アンケート'
  description text,                         -- 説明文
  reward_stamps integer not null default 3, -- 報酬スタンプ数（pt単位: 3 = 0.3個）
  is_active boolean not null default true,  -- 公開中かどうか
  start_date timestamptz,                   -- 公開開始日
  end_date timestamptz,                     -- 公開終了日
  created_at timestamptz not null default now()
);

-- ========================================
-- 2. survey_answers テーブル（回答データ）
-- ========================================
create table survey_answers (
  id uuid primary key default gen_random_uuid(),
  user_id text not null references profiles(id) on delete cascade,
  survey_id text not null references surveys(id) on delete cascade,

  -- 質問回答（Phase 1: 固定スキーマ）
  q1_rating integer,        -- Q1: 5段階評価（1〜5）
  q2_comment text,          -- Q2: 自由記述
  q3_recommend integer,     -- Q3: 推奨度（0〜10、NPS計算用）

  created_at timestamptz not null default now(),

  -- 重複回答防止
  unique(user_id, survey_id)
);

-- インデックス
create index idx_survey_answers_survey_id on survey_answers(survey_id);
create index idx_survey_answers_created_at on survey_answers(created_at);
create index idx_survey_answers_user_id on survey_answers(user_id);

-- ========================================
-- 3. survey_targets テーブル（配信対象者）
-- ========================================
create table survey_targets (
  id uuid primary key default gen_random_uuid(),
  user_id text not null references profiles(id) on delete cascade,
  survey_id text not null references surveys(id) on delete cascade,

  -- 表示制御
  show_on_liff_open boolean not null default false,  -- LIFFアプリ初期表示するか

  -- 回答状況
  answered_at timestamptz,  -- 回答完了日時（NULLなら未回答）

  -- 後回しカウント
  postponed_count integer not null default 0,  -- 「あとで」を押した回数
  last_postponed_at timestamptz,              -- 最後に「あとで」を押した日時

  -- メタ情報
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),

  -- 重複防止
  unique(user_id, survey_id)
);

-- インデックス
create index idx_survey_targets_user_id on survey_targets(user_id);
create index idx_survey_targets_survey_id on survey_targets(survey_id);
create index idx_survey_targets_show_on_liff on survey_targets(show_on_liff_open) where show_on_liff_open = true;
create index idx_survey_targets_answered on survey_targets(answered_at);

-- ========================================
-- 4. RLSポリシー
-- ========================================
alter table surveys enable row level security;
alter table survey_answers enable row level security;
alter table survey_targets enable row level security;

-- surveys: 公開中のアンケートは全員が参照可能
create policy "anon_can_read_active_surveys"
  on surveys for select
  using (is_active = true);

-- survey_answers: 自分の回答のみ参照・挿入可能
create policy "anon_can_read_own_answers"
  on survey_answers for select
  using (true);  -- アプリ層で user_id フィルタ

create policy "anon_can_insert_own_answers"
  on survey_answers for insert
  with check (true);  -- アプリ層で user_id 検証

-- survey_targets: 自分のターゲット情報のみ参照・更新可能
create policy "anon_can_read_own_targets"
  on survey_targets for select
  using (true);  -- アプリ層で user_id フィルタ

create policy "anon_can_update_own_targets"
  on survey_targets for update
  using (true);  -- アプリ層で user_id フィルタ

-- 管理者（service_role）は全件操作可能（RLSをバイパス）

-- ========================================
-- 5. ビュー: 配信対象者の一覧（管理画面用）
-- ========================================
create or replace view survey_target_status as
select
  st.id,
  st.survey_id,
  s.title as survey_title,
  st.user_id,
  p.real_name,
  p.display_name,  -- ★ line_display_name ではなく display_name
  st.show_on_liff_open,
  st.answered_at,
  case
    when st.answered_at is not null then '回答済み'
    else '未回答'
  end as status,
  st.postponed_count,
  st.last_postponed_at,
  st.created_at
from survey_targets st
join surveys s on st.survey_id = s.id
join profiles p on st.user_id = p.id
order by st.created_at desc;

-- ========================================
-- 6. RPC関数: 「あとで」カウント増加
-- ========================================
create or replace function increment_survey_postponed(
  p_user_id text,
  p_survey_id text
)
returns void as $$
begin
  update survey_targets
  set
    postponed_count = postponed_count + 1,
    last_postponed_at = now(),
    updated_at = now()
  where user_id = p_user_id
    and survey_id = p_survey_id;
end;
$$ language plpgsql security definer;

-- ========================================
-- 7. 初期データ
-- ========================================
insert into surveys (id, title, description, reward_stamps) values
  ('satisfaction_2026Q1', 'ご利用満足度アンケート', '当院のサービスについてお聞かせください', 3);
```

## 9. 実装上の補足事項（2026-02-26 追記）

### 9.1 Supabase クライアント名の確認（ダッシュボード側の最新状況）

**重要:** 本プロジェクトの実際の構成は以下の通りです。

- **管理用（service_role）:** `createSupabaseAdminClient()` を **`@/lib/supabase/server-admin`** から import。  
  `app/api/families`・`app/api/activity-logs`・`app/api/auth` 等で使用。
- **LIFF側（患者）:** anon key のクライアント（プロジェクトで利用している Supabase クライアント）で、`user_id` は **`profiles.id`** を指定する。

**ダッシュボード用 API の記述例:**
```typescript
import { createSupabaseAdminClient } from '@/lib/supabase/server-admin';
const supabase = createSupabaseAdminClient();
```

### 9.2 LIFF側の認証方式

**重要:** 本プロジェクトは **Supabase Auth を使用していません**（LINE ログインのみ）。

- LINE SDK の `userId` は **`profiles.line_user_id`** に相当する。
- **`survey_targets.user_id`** は **`profiles.id`**（PK）を参照するため、LIFF 側では「LINE user id → profiles.id」の解決が必要。
- 解決例: (1) 既存 API で `line_user_id` を渡し `profiles.id` を返す、(2) anon で `profiles` を `line_user_id = ?` で取得し `id` を使用。

**実装例（profiles.id を既に保持している場合）:**
```typescript
const { profile } = useLiff(); // アプリで profile.id = profiles.id を保持している想定

const { data } = await supabase
  .from('survey_targets')
  .eq('user_id', profile.id)  // ★ profiles.id（LINE userId の場合は事前に解決すること）
  .select('*');
```

### 9.4 データベーススキーマ上の注意点

**ビューの ORDER BY について:**
- PostgreSQLの仕様上、ビュー定義内の `ORDER BY` は結果順序を保証しません
- API実装時に必ず `.order()` を明示的に指定してください
