## 現在の問題点

### 🔴 Critical: 誰でも他人のデータにアクセス可能

**現在のRLSポリシー：**
```sql
-- すべてのユーザーが全データにアクセス可能
CREATE POLICY "allow_public_read"
  ON profiles FOR SELECT
  USING (true);  -- ⚠️ 常にtrue = 制限なし
```

**問題の原因：**
1. LINE LIFF認証を使用している
2. Supabase Authを使用していない
3. RLSポリシーで「現在のユーザー」を識別できない

**結果：**
- ❌ 他人のプロフィールを閲覧可能
- ❌ 他人のスタンプ数を変更可能
- ❌ 他人のスタンプ履歴を削除可能

## 選択肢Aの概要

### 🎯 基本方針

**Supabase Auth + LINE連携で、RLSポリシーのみで完全なセキュリティを実現**

```
LINE LIFF認証
  ↓
Supabase Auth にサインイン（カスタムクレームでLINE User IDを追加）
  ↓
auth.jwt()->>'line_user_id' が使えるようになる
  ↓
RLSポリシーで「自分のデータのみ」を判定可能
  ↓
既存のクエリがそのまま動作（RLSが自動的に制限）
```

### ✅ メリット

1. **最小限のコード変更**
   - 変更ファイル：1ファイル（`hooks/useLiff.ts`）
   - 既存のクエリはそのまま動作

2. **完全なセキュリティ**
   - 他人のデータ閲覧：不可
   - 他人のデータ変更：不可
   - 他人のデータ削除：不可

3. **Supabaseの標準機能を活用**
   - `auth.jwt()`を使った標準的なRLSポリシー
   - 公式ドキュメントが豊富
   - メンテナンス性が高い

4. **短い実装期間**
   - 工数：2-3日
   - リスク：低

## 実装手順

### ステップ1：認証フローの修正

#### 現在のコード（hooks/useLiff.ts）

```typescript
// LINE LIFFでログイン
const lineProfile = await liff.getProfile();
const lineUserId = lineProfile.userId;

// 🔴 Supabase Authを使っていない
const { data: profile } = await supabase
  .from("profiles")
  .select("*")
  .eq("id", lineUserId)
  .single();
```

#### 修正後のコード

```typescript
import { supabase } from "@/lib/supabase";

// LINE LIFFでログイン
const lineProfile = await liff.getProfile();
const lineUserId = lineProfile.userId;
const lineAccessToken = liff.getAccessToken();

// ✅ Supabaseにカスタムトークンでサインイン
const { data: authData, error: authError } = await supabase.auth.signInWithIdToken({
  provider: 'line',
  token: lineAccessToken,
  options: {
    data: {
      line_user_id: lineUserId,  // カスタムクレームに追加
    }
  }
});

if (authError) {
  console.error('Supabase Auth error:', authError);
  throw authError;
}

// これで auth.uid() と auth.jwt()->>'line_user_id' が使えるようになる
const { data: profile } = await supabase
  .from("profiles")
  .select("*")
  .single();  // ✅ RLSポリシーが自動的に自分のデータのみ返す
```

### ステップ2：RLSポリシーの設定

新しいマイグレーションファイルを作成：

#### supabase/024_auth_based_rls.sql

```sql
-- =====================================
-- Supabase Auth + LINE連携用RLSポリシー
-- =====================================

-- =====================================
-- profiles テーブル
-- =====================================

-- 既存のポリシーを削除
DROP POLICY IF EXISTS "allow_public_read" ON profiles;
DROP POLICY IF EXISTS "allow_public_insert" ON profiles;
DROP POLICY IF EXISTS "allow_public_update" ON profiles;

-- SELECT: 自分のプロフィールのみ閲覧可能
CREATE POLICY "auth_users_can_read_own_profile"
  ON profiles FOR SELECT
  TO authenticated, anon
  USING (
    line_user_id = auth.jwt()->>'line_user_id'
  );

-- INSERT: 自分のプロフィールのみ作成可能
CREATE POLICY "auth_users_can_insert_own_profile"
  ON profiles FOR INSERT
  TO authenticated, anon
  WITH CHECK (
    line_user_id = auth.jwt()->>'line_user_id'
  );

-- UPDATE: 自分のプロフィールのみ更新可能
-- ただし、real_name と ticket_number は変更不可
CREATE POLICY "auth_users_can_update_own_profile"
  ON profiles FOR UPDATE
  TO authenticated, anon
  USING (
    line_user_id = auth.jwt()->>'line_user_id'
  )
  WITH CHECK (
    line_user_id = auth.jwt()->>'line_user_id'
    AND (real_name IS NULL OR real_name = OLD.real_name)
    AND (ticket_number IS NULL OR ticket_number = OLD.ticket_number)
  );

-- DELETE: 誰も削除できない（管理ダッシュボードのみ）
CREATE POLICY "deny_delete_profiles"
  ON profiles FOR DELETE
  TO authenticated, anon
  USING (false);

-- =====================================
-- stamp_history テーブル
-- =====================================

-- 既存のポリシーを削除
DROP POLICY IF EXISTS "allow_public_read" ON stamp_history;
DROP POLICY IF EXISTS "allow_public_insert" ON stamp_history;
DROP POLICY IF EXISTS "allow_public_delete" ON stamp_history;

-- SELECT: 自分のスタンプ履歴のみ閲覧可能
CREATE POLICY "auth_users_can_read_own_history"
  ON stamp_history FOR SELECT
  TO authenticated, anon
  USING (
    user_id IN (
      SELECT id FROM profiles WHERE line_user_id = auth.jwt()->>'line_user_id'
    )
  );

-- INSERT: 自分のスタンプ履歴のみ追加可能
CREATE POLICY "auth_users_can_insert_own_history"
  ON stamp_history FOR INSERT
  TO authenticated, anon
  WITH CHECK (
    user_id IN (
      SELECT id FROM profiles WHERE line_user_id = auth.jwt()->>'line_user_id'
    )
  );

-- UPDATE: 誰も更新できない
CREATE POLICY "deny_update_stamp_history"
  ON stamp_history FOR UPDATE
  TO authenticated, anon
  USING (false);

-- DELETE: 誰も削除できない（管理ダッシュボードのみ）
CREATE POLICY "deny_delete_stamp_history"
  ON stamp_history FOR DELETE
  TO authenticated, anon
  USING (false);

-- =====================================
-- reward_exchanges テーブル
-- =====================================

-- 既存のポリシーを削除
DROP POLICY IF EXISTS "allow_public_read_exchanges" ON reward_exchanges;
DROP POLICY IF EXISTS "allow_public_insert_exchanges" ON reward_exchanges;

-- SELECT: 自分の交換履歴のみ閲覧可能
CREATE POLICY "auth_users_can_read_own_exchanges"
  ON reward_exchanges FOR SELECT
  TO authenticated, anon
  USING (
    user_id IN (
      SELECT id FROM profiles WHERE line_user_id = auth.jwt()->>'line_user_id'
    )
  );

-- INSERT: 自分の名前でのみ交換可能
CREATE POLICY "auth_users_can_insert_own_exchanges"
  ON reward_exchanges FOR INSERT
  TO authenticated, anon
  WITH CHECK (
    user_id IN (
      SELECT id FROM profiles WHERE line_user_id = auth.jwt()->>'line_user_id'
    )
  );

-- UPDATE: 誰も更新できない
CREATE POLICY "deny_update_reward_exchanges"
  ON reward_exchanges FOR UPDATE
  TO authenticated, anon
  USING (false);

-- DELETE: 誰も削除できない
CREATE POLICY "deny_delete_reward_exchanges"
  ON reward_exchanges FOR DELETE
  TO authenticated, anon
  USING (false);

-- =====================================
-- patient_dental_records テーブル
-- =====================================

-- 既存のポリシーを削除
DROP POLICY IF EXISTS "anon_can_read_own_dental_records" ON patient_dental_records;

-- SELECT: 自分の記録のみ閲覧可能
CREATE POLICY "auth_users_can_read_own_dental_records"
  ON patient_dental_records FOR SELECT
  TO authenticated, anon
  USING (
    patient_id IN (
      SELECT id FROM profiles WHERE line_user_id = auth.jwt()->>'line_user_id'
    )
  );

-- INSERT: 誰も追加できない（管理ダッシュボードのみ）
CREATE POLICY "deny_insert_dental_records"
  ON patient_dental_records FOR INSERT
  TO authenticated, anon
  WITH CHECK (false);

-- UPDATE: 誰も更新できない（管理ダッシュボードのみ）
CREATE POLICY "deny_update_dental_records"
  ON patient_dental_records FOR UPDATE
  TO authenticated, anon
  USING (false);

-- DELETE: 誰も削除できない（管理ダッシュボードのみ）
CREATE POLICY "deny_delete_dental_records"
  ON patient_dental_records FOR DELETE
  TO authenticated, anon
  USING (false);

-- =====================================
-- event_logs テーブル
-- =====================================

-- INSERT: 誰でも記録可能（ログ用）
CREATE POLICY "anyone_can_insert_event_logs"
  ON event_logs FOR INSERT
  TO authenticated, anon
  WITH CHECK (true);

-- SELECT/UPDATE/DELETE: 誰もアクセスできない（管理ダッシュボードのみ）
CREATE POLICY "deny_read_event_logs"
  ON event_logs FOR SELECT
  TO authenticated, anon
  USING (false);
```

### ステップ3：管理操作はSERVICE_ROLE_KEY経由

スタッフ操作（削除、管理者権限での追加など）は、既存の**サーバーサイドAPI Route**で`SERVICE_ROLE_KEY`を使用：

```typescript
// app/api/staff/delete-stamp/route.ts（既存）

import { getSupabaseAdmin } from "@/lib/supabase";

export async function POST(request: Request) {
  // スタッフPIN認証済み

  const supabaseAdmin = getSupabaseAdmin();  // SERVICE_ROLE_KEY

  // ✅ 管理者権限でDELETE可能（RLSをバイパス）
  const { error } = await supabaseAdmin
    .from("stamp_history")
    .delete()
    .eq("id", stampId);

  // ...
}
```

**重要：** 管理ダッシュボード側は`SERVICE_ROLE_KEY`を使用しているため、**RLSポリシーの影響を受けません**。

**既存のクエリはすべてそのまま動作します！**

```typescript
// これまで通りのコード
const { data } = await supabase
  .from("stamp_history")
  .select("*");

// ✅ RLSポリシーが自動的に「自分のデータのみ」に制限してくれる
```

**理由：**
- RLSポリシーは**データベース側で自動的に適用**される
- クライアント側のコードは変更不要
- `.eq("id", userId)` などのフィルタも引き続き機能
