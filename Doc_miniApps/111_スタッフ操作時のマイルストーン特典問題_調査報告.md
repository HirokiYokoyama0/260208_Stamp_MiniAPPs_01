## 📋 報告された問題

### 問題1: スタンプ減少時に特典が「受付確認中」のまま残る

**症状:**
- スタッフ操作でスタンプ数を0に減らす
- 10スタンプのマイルストーン特典が「受付確認中」(pending) のまま残る
- **期待:** 特典が削除される、または status が cancelled になる
- **実際:** pending のまま表示される

### 問題2: スタンプ増加時にマイルストーン特典が自動付与される

**症状:**
- 管理ダッシュボードでスタンプ数を増やす
- マイルストーン特典が「受付確認中」(pending) で自動付与される
- **期待:** スタッフ操作では自動付与されない（手動で付与する運用を想定）
- **実際:** 自動付与されてしまう

### 問題3: スタンプを0にしたときにマイルストーン特典が消えない

**症状:**
- スタッフ操作でスタンプ数を0にする
- マイルストーン特典が削除されない
- **期待:** 0個に減らした場合、全てのマイルストーン特典が削除される
- **実際:** 特典が残る

## 🔍 調査結果

### 1. 現在の実装の動作

#### スタッフ操作API（ミニアプリ側）: `app/api/stamps/manual/route.ts`

**処理フロー:**
```typescript
1. スタンプ数を取得 (currentStampCount)
2. 新しいスタンプ数を設定 (newStampCount)
3. changeAmount = newStampCount - currentStampCount を計算
4. stamp_history の全レコードを削除（起点としてリセット）
5. stamp_history に新しいレコードを INSERT (stamp_number = newStampCount)
6. profiles.stamp_count を直接 UPDATE (stamp_number = newStampCount)
7. changeAmount < 0 の場合のみ:
   - invalidateMilestoneRewards(userId, currentStampCount, newStampCount) を呼ぶ
8. changeAmount > 0 の場合:
   - 何もしない（マイルストーン判定なし、特典付与なし）
```

**重要な点:**
- スタンプ**減少時**: `invalidateMilestoneRewards()` が呼ばれる
- スタンプ**増加時**: マイルストーン判定も特典付与も**されない**

### 2. invalidateMilestoneRewards() の実装確認

#### `lib/milestones.ts:305-364`

**処理フロー:**
```typescript
1. newStampCount >= oldStampCount の場合は何もしない（増加時はスキップ）
2. 削除すべきマイルストーンを計算
   - new10 = floor(newStampCount / 10)
   - old10 = floor(oldStampCount / 10)
   - milestoneArray = [(new10+1)*10, (new10+2)*10, ..., old10*10]
3. reward_exchanges テーブルを UPDATE:
   - status を 'cancelled' に変更
   - user_id, is_milestone_based=true, milestone_reached IN milestoneArray
   - status IN ['pending', 'approved', 'completed'] のみ（'used'は除外）
4. 変更件数を返す
```

**テスト結果:**
- ✅ ANON_KEYでUPDATEは成功する（RLSポリシーで許可されている）
- ✅ テストで `pending` → `cancelled` への変更は可能

### 3. 問題の原因分析

#### 問題1・3: スタンプ減少時に特典が残る

**原因の可能性:**

**A. invalidateMilestoneRewards() の計算ロジックの問題**

例: 15個 → 0個に減らす場合
```typescript
old10 = floor(15 / 10) = 1
new10 = floor(0 / 10) = 0

milestoneArray = [(0+1)*10] = [10]
```
✅ 10が正しく含まれる → 問題なし

**B. UPDATE文の条件の問題**

`invalidateMilestoneRewards()` の UPDATE文:
```typescript
.eq('user_id', userId)
.eq('is_milestone_based', true)
.in('milestone_reached', milestoneArray)
.in('status', ['pending', 'approved', 'completed'])
```

テストユーザーの特典データ:
- milestone_reached: 10
- is_milestone_based: true
- status: pending
- user_id: U5c70cd61f4fe89a65381cd7becee8de3

→ ✅ 全ての条件に一致する → UPDATEされるはず

**C. 実行されているか確認が必要**

可能性:
1. `invalidateMilestoneRewards()` が実際に呼ばれていない？
2. エラーが発生しているが、try-catchでキャッチされている？
3. invalidatedCount = 0 が返っている（= 条件に一致するレコードがない）？

#### 問題2: スタンプ増加時にマイルストーン特典が自動付与される

**ミニアプリ側の実装確認:**
- `app/api/stamps/manual/route.ts` では `changeAmount > 0` の場合、何もしない
- `checkMilestones()` も `grantMilestoneReward()` も呼ばれない

**結論:**
- ❌ ミニアプリ側のスタッフ操作APIでは自動付与されない
- ⚠️ 管理ダッシュボード側で別のAPIを使用している可能性が高い

**推測される原因:**
- 管理ダッシュボード側で `PUT /api/stamps/manual` とは別のエンドポイントを使用
- または、管理ダッシュボード側に独自のスタンプ数変更機能がある
- そちらでマイルストーン判定と特典付与を実装している可能性

## 🔧 修正案

### 修正1: invalidateMilestoneRewards() にログを追加

**目的:** 実際に関数が呼ばれているか、何件UPDATEされたか確認

**変更箇所:** `lib/milestones.ts:305-364`

```typescript
export async function invalidateMilestoneRewards(
  userId: string,
  oldStampCount: number,
  newStampCount: number
): Promise<number> {
  // 増加時は何もしない
  if (newStampCount >= oldStampCount) {
    console.log(`⏭️ [invalidateMilestoneRewards] スキップ: スタンプ増加 (${oldStampCount} → ${newStampCount})`);
    return 0;
  }

  // 削除すべきマイルストーンを計算
  const old10 = Math.floor(oldStampCount / 10);
  const new10 = Math.floor(newStampCount / 10);
  const milestoneArray: number[] = [];

  for (let i = new10 + 1; i <= old10; i++) {
    milestoneArray.push(i * 10);
  }

  if (milestoneArray.length === 0) {
    console.log(`⏭️ [invalidateMilestoneRewards] 無効化するマイルストーンなし (${oldStampCount} → ${newStampCount})`);
    return 0;
  }

  console.log(`🔄 [invalidateMilestoneRewards] 開始:`, {
    userId: userId.substring(0, 8) + '...',
    oldStampCount,
    newStampCount,
    milestonesToInvalidate: milestoneArray
  });

  // ← 🆕 UPDATE前に対象レコードを確認
  const { data: targetRewards } = await supabase
    .from('reward_exchanges')
    .select('id, milestone_reached, status')
    .eq('user_id', userId)
    .eq('is_milestone_based', true)
    .in('milestone_reached', milestoneArray)
    .in('status', ['pending', 'approved', 'completed']);

  console.log(`📋 [invalidateMilestoneRewards] UPDATE対象:`, {
    count: targetRewards?.length || 0,
    rewards: targetRewards?.map(r => `milestone:${r.milestone_reached}, status:${r.status}`)
  });

  // reward_exchanges をソフトデリート（status = 'cancelled'）
  const { data, error } = await supabase
    .from('reward_exchanges')
    .update({
      status: 'cancelled',
      notes: `スタッフ操作により無効化 (${oldStampCount} → ${newStampCount})`,
      updated_at: new Date().toISOString()
    })
    .eq('user_id', userId)
    .eq('is_milestone_based', true)
    .in('milestone_reached', milestoneArray)
    .in('status', ['pending', 'approved', 'completed'])
    .select('id, milestone_reached, status');

  if (error) {
    console.error('❌ [invalidateMilestoneRewards] UPDATE失敗:', error);
    throw new Error(`Failed to invalidate rewards: ${error.message}`);
  }

  const invalidatedCount = data?.length || 0;

  console.log(`✅ [invalidateMilestoneRewards] 完了:`, {
    userId: userId.substring(0, 8) + '...',
    invalidatedCount,
    updatedRecords: data?.map(r => `id:${r.id.substring(0, 8)}, milestone:${r.milestone_reached}`)
  });

  return invalidatedCount;
}
```

**質問:**
- 管理ダッシュボードでスタンプ数を変更した際、どのAPIエンドポイントを使用していますか？
- ミニアプリ側の `POST /api/stamps/manual` を使用していますか？
- それとも、管理ダッシュボード専用のAPIエンドポイントがありますか？

**背景:**
- ミニアプリ側の `POST /api/stamps/manual` では、スタンプ増加時にマイルストーン特典を自動付与しない実装になっている
- しかし、管理ダッシュボードでスタンプ数を増やすと特典が自動付与されるという報告がある
- 別のAPIエンドポイントを使用している可能性がある

### 2. 修正方針の共有

**ミニアプリ側の方針:**
- スタンプ減少時: マイルストーン特典を自動的に `cancelled` にする
- スタンプ増加時: マイルストーン特典を自動付与**しない**

**管理ダッシュボード側への要望:**
- スタッフ操作でスタンプ数を変更した場合、マイルストーン特典を自動付与しない
- 必要であれば、管理ダッシュボード側で手動で特典を付与する機能を追加

### 3. 仕様確認

**質問:**
- スタッフ操作でスタンプ数を増やした場合、マイルストーン特典を自動付与すべきですか？
- それとも、手動で付与する運用にすべきですか？

**推奨:**
- **手動付与**: スタッフが意図的に特典を付与する（管理ダッシュボードで操作）
- **理由**: スタッフ操作は特殊な状況（エラー修正、クレーム対応など）が多いため、自動付与は不適切
