# スタンプ問題：原因候補の優先順位付け

**作成日**: 2026-03-28
**状況**: Android 1台で正常動作確認済み、複数のユーザーから問題報告あり

## 📋 報告されている問題（再整理）

### 問題1: 15スタンプQRが10ポイントしか付与されない
- 優良患者様用QR（15スタンプ想定）をスキャン → 10ポイントのみ付与
- **別の携帯では15ポイント正常に付与された** ← 同じQRコードで結果が異なる

### 問題2: 購買インセンティブQRが1度しかスキャンできない
- 購買インセンティブQRコードが1度のみしかスキャンできなかった
- 期待：回数制限なくスキャン可能

## 🔍 確認済みの事実

| 項目 | 状態 |
|------|------|
| 開発者のAndroidスマホ | ✅ 正常動作 |
| コード実装（API） | ✅ 正しい |
| コード実装（フロントエンド） | ✅ 正しい |
| 購買インセンティブの1日1回制限除外 | ✅ 正しく実装済み |
| デバッグログ | ✅ 既に実装済み（コンソールログ） |
| イベントログ（Supabase） | ✅ 今回実装済み |

## 🎯 原因候補（優先順位付け）

### 【最優先】候補1: QRコード自体の問題

#### 可能性：⭐⭐⭐⭐⭐ 非常に高い

**理由**:
- 「別の携帯では15ポイント正常に付与された」→ **同じQRコードのはずが違う結果**
- コード実装は正しい
- Android 1台では問題ないが、複数ユーザーで問題発生

**仮説**:
1. **院内に複数のQRコードが混在している**
   - 優良患者様用（15スタンプ）のつもりで設置
   - 実際には通常患者様用（10スタンプ）が混ざっている
   - ユーザーがどちらをスキャンするかで結果が変わる

2. **QRコード生成時のミス**
   - 管理ダッシュボード側で生成時に間違ったペイロードが混入
   - 例: 優良患者様用のラベルで `{"type":"premium","stamps":10}` を生成

3. **QRコード画像の取り違え**
   - PDF生成時やプリント時に別のQRコードが混入

**確認方法**:
```
【優先度：最高】
1. 管理ダッシュボード開発者に確認
   - 優良患者様用QRコードのペイロードは？
   - 通常患者様用QRコードのペイロードは？
   - 購買インセンティブQRコードのペイロードは？

2. 院内のQRコード現物確認
   - 複数の場所に設置されているQRコードを全てスキャン
   - それぞれのペイロードを記録
   - 想定と一致するか確認

3. 問題が発生したユーザーに確認
   - どこに設置されていたQRコードをスキャンしたか
   - （可能であれば）そのQRコードをもう一度スキャンしてもらう
```

### 【高優先】候補2: デバイスや環境による LIFF の挙動差

#### 可能性：⭐⭐⭐ 中程度

**理由**:
- LIFF SDK の `scanCodeV2` は、iOS/Android でネイティブカメラを呼び出す
- ネイティブカメラの QR 読み取り精度はデバイスに依存
- LINE アプリのバージョンによって挙動が異なる可能性

**仮説**:
1. **QRコードの読み取りエラー**
   - カメラの読み取り精度が低く、QRコードを誤読
   - 例: `{"type":"premium","stamps":15}` → `{"type":"premium","stamps":10}` に誤読
   - ただし、**JSONパースエラーにならない範囲での誤読は非常に稀**

2. **LINEアプリのバージョン差**
   - 古いLINEアプリで `scanCodeV2` の挙動が異なる
   - iOS/Androidの特定バージョンでバグがある

**確認方法**:
```
【優先度：高】
1. イベントログの qr_raw_value を確認
   - Supabase の event_logs テーブル
   - SELECT metadata->>'qr_raw_value' FROM event_logs
     WHERE event_name = 'stamp_scan_api_request'
   → 生値が正しいか確認

2. 複数デバイスでテスト
   - 開発者以外のAndroidスマホでもテスト
   - iPhoneでもテスト
   - 同じQRコードで全て同じ結果になるか確認

3. ユーザーのLINEアプリバージョン確認
   - 問題が発生したユーザーのLINEバージョンを確認
   - 最新版にアップデートして再度試してもらう
```

---

### 【中優先】候補3: 購買インセンティブの問題 - ユーザーの誤解または別のエラー

#### 可能性：⭐⭐⭐ 中程度

**理由**:
- コード上は1日1回制限から除外されている
- しかし「1度しかスキャンできなかった」と報告あり

**仮説**:
1. **実は複数回スキャンできているが気付いていない**
   - スタンプ履歴を確認していない
   - 2回目以降も成功しているが、UIの表示が分かりにくい

2. **別のエラーメッセージが出ている**
   - 例: 「無効なQRコードです」（JSONパースエラー）
   - 1日1回制限ではなく、QRコード自体が読み取れていない

3. **QRコードが実は購買インセンティブ用ではない**
   - 通常患者様用QRを誤って使っている
   - → 1日1回制限が効く

**確認方法**:
```
【優先度：中】
1. イベントログで購買インセンティブの記録を確認
   - SELECT * FROM event_logs
     WHERE event_name = 'stamp_scan_api_request'
       AND metadata->'qr_parsed'->>'type' = 'purchase'
   → 本当に購買インセンティブQRをスキャンしているか

2. 同じユーザーが複数回スキャンしているか確認
   - SELECT user_id, COUNT(*) FROM event_logs
     WHERE event_name = 'stamp_scan_success'
       AND metadata->>'type' = 'purchase'
     GROUP BY user_id
   → 複数回記録されているか

3. ユーザーにヒアリング
   - どのQRコードをスキャンしたか
   - エラーメッセージは何と表示されたか
   - スタンプ履歴タブで5個ずつ増えているか確認してもらう
```

---

### 【低優先】候補4: データベーストリガーの問題

#### 可能性：⭐ 非常に低い

**理由**:
- トリガーは単純な MAX(stamp_number) を計算するだけ
- Android 1台では正常動作している
- スタンプ数は減っている（問題報告より）→ トリガーは動いている

**仮説**:
- `amount` カラムがNULLになり、デフォルト値10が適用される
- ただし、APIコードで必ず `stamps` を設定しているため**ほぼありえない**

**確認方法**:
```
【優先度：低】
- stamp_history テーブルの amount カラムを直接確認
- SELECT * FROM stamp_history WHERE amount IS NULL OR amount != 15
  → amount が意図通りか確認
```

---

### 【低優先】候補5: ネットワークやタイミングの問題

#### 可能性：⭐ 非常に低い

**理由**:
- ネットワークエラーなら「サーバーエラー」が表示されるはず
- タイミング問題なら再現性が不安定すぎる

**仮説**:
- API リクエストが途中で切れて、部分的なデータが送信される
- → しかし、HTTPリクエストは全て送信されるか全て失敗するかのどちらか

### 2. 【高優先】イベントログの確認（Supabase）

**担当**: 開発者

**確認SQL**:
```sql
-- 1. 最近のQRスキャンAPIリクエストを確認（QR生値を見る）
SELECT
  created_at,
  user_id,
  metadata->>'qr_raw_value' as qr_raw,
  metadata->'qr_parsed'->>'type' as type,
  metadata->'qr_parsed'->>'stamps' as stamps
FROM event_logs
WHERE event_name = 'stamp_scan_api_request'
  AND created_at >= current_date - interval '7 days'
ORDER BY created_at DESC
LIMIT 50;

-- 2. スタンプ個数不一致を検出
WITH requests AS (
  SELECT
    user_id,
    created_at,
    (metadata->'request_payload'->>'stamps')::int as requested
  FROM event_logs
  WHERE event_name = 'stamp_scan_api_request'
),
successes AS (
  SELECT
    user_id,
    created_at,
    (metadata->>'stamps_added')::int as actual
  FROM event_logs
  WHERE event_name = 'stamp_scan_success'
)
SELECT
  req.user_id,
  req.created_at,
  req.requested,
  suc.actual,
  (req.requested - suc.actual) as 差分
FROM requests req
JOIN successes suc
  ON req.user_id = suc.user_id
  AND suc.created_at BETWEEN req.created_at AND req.created_at + interval '5 seconds'
WHERE req.requested != suc.actual;  -- 不一致のみ

-- 3. 購買インセンティブの複数回スキャン確認
SELECT
  user_id,
  COUNT(*) as scan_count,
  MIN(created_at) as first_scan,
  MAX(created_at) as last_scan
FROM event_logs
WHERE event_name = 'stamp_scan_success'
  AND metadata->>'type' = 'purchase'
  AND created_at >= current_date - interval '7 days'
GROUP BY user_id
HAVING COUNT(*) > 1;
```

**根拠**:
