# スタンプ問題：15スタンプQRのみの状況での原因分析

**作成日**: 2026-03-28
**重要な前提**: **院内には15スタンプQRしか存在していない**

## 📋 前提条件の確認

| 項目 | 状態 |
|------|------|
| 15スタンプQR（優良患者様用） | ✅ 運用中 |
| 10スタンプQR（通常患者様用） | ❌ 未展開 |
| 5スタンプQR（購買インセンティブ） | ❌ 未展開または限定的 |

→ **QRコードの混在・取り違えは除外できる**

## 📋 報告されている問題（再確認）

### 問題1: 15スタンプQRが10ポイントしか付与されない
- 15スタンプQRをスキャン → **10ポイントのみ付与**
- **別の携帯では15ポイント正常に付与された** ← 同じQRコードで結果が異なる
- 開発者のAndroidスマホでは正常動作

### 問題2: 購買インセンティブQRが1度しかスキャンできない
- （この問題は一旦保留、問題1に集中）

## 🎯 原因候補（再評価）

### 【最優先】候補1: QRコード自体のペイロードが間違っている

#### 可能性：⭐⭐⭐⭐⭐ 非常に高い

**仮説**:
管理ダッシュボード側で15スタンプQRを生成したつもりが、実際には以下のようなペイロードになっている：

```json
// 期待値
{"type":"premium","stamps":15}

// 実際の値（可能性1）
{"type":"premium","stamps":10}

// 実際の値（可能性2）
{"type":"regular","stamps":10}
```

**「別の携帯では15ポイント正常に付与された」の説明**:
- 実は院内に**複数の15スタンプQRコード画像**が存在
- 最初に生成したQRコード（テスト用？）は `stamps:10` だった
- 後で修正して再生成したQRコードは `stamps:15` になった
- 院内のどこかに**古いQRコード**が残っている
- ユーザーがどちらをスキャンするかで結果が変わる

**確認方法**:
1. **管理ダッシュボード開発者に確認**
   - 15スタンプQRコードのペイロードを見せてもらう
   - 実際の値は？

2. **院内の全QRコードをスキャンして確認**
   - 受付、待合室、診察室など
   - それぞれのペイロードを記録

3. **イベントログで確認**（既に実装済み）
   ```sql
   SELECT
     created_at,
     metadata->>'qr_raw_value' as qr_raw,
     metadata->'qr_parsed'->>'stamps' as stamps
   FROM event_logs
   WHERE event_name = 'stamp_scan_api_request'
   ORDER BY created_at DESC
   LIMIT 20;
   ```

### 【高優先】候補2: デバイスや環境によるQRコード読み取りエラー

#### 可能性：⭐⭐⭐ 中程度

**仮説**:
- LIFF の `scanCodeV2` がQRコードを誤読している
- 例: `{"type":"premium","stamps":15}` → `{"type":"premium","stamps":10}` に誤読
- カメラの性能、照明、QRコードの印刷品質などに依存

**しかし、この可能性は低い理由**:
1. JSONパースエラーにならない範囲での誤読は**非常に稀**
2. `15` → `10` という変化は、**1文字だけ変わる**のではなく**2文字が変わる**
   - `15` の ASCII: `31 35`
   - `10` の ASCII: `31 30`
   - `5` → `0` の1文字だけ変わるのは偶然すぎる

3. もし誤読なら、むしろ以下のようなエラーになるはず：
   - `stamps:1` や `stamps:1511` など意味不明な値
   - JSONパースエラー（閉じ括弧が消えるなど）

**確認方法**:
- イベントログの `qr_raw_value` を確認
- 誤読されている場合、生値が壊れているはず

---

### 【中優先】候補3: クライアント側のコードに問題がある

#### 可能性：⭐⭐ 低い

**仮説**:
- `app/scan/page.tsx` の `processQRCode` 関数でパース時に何か問題がある

**しかし、この可能性も低い理由**:
1. コードレビュー済み - パース処理は単純な `JSON.parse(qrValue)`
2. 開発者のAndroidでは正常動作している
3. 「別の携帯では15ポイント正常」→ 同じコードで違う結果は説明できない

**念のため確認**:

## 📊 最も可能性が高いシナリオ

### 🎯 結論：QRコード自体のペイロードが間違っている

**理由**:
1. 15スタンプQRしか存在しないのに10ポイントが付与される
2. 「別の携帯では15ポイント」→ 違うQRコードをスキャンしている
3. コード実装は全て正しい

**最も可能性が高い状況**:

```
【シナリオ】
1. 最初にテスト用15スタンプQRを生成
   → 実はペイロードが {"type":"premium","stamps":10} だった
   → PDF生成、プリント、院内設置（場所A）

2. その後、正しい15スタンプQRを再生成
   → ペイロード {"type":"premium","stamps":15}
   → PDF生成、プリント、院内設置（場所B）

3. 現在、院内に2種類のQRコードが混在
   - 場所A: 10ポイントQR（古い）
   - 場所B: 15ポイントQR（新しい）

4. ユーザーがどちらをスキャンするかで結果が変わる
   - 場所Aをスキャン → 10ポイント
   - 場所Bをスキャン → 15ポイント
```

## 🚀 次のアクション（最優先）

### 1. 管理ダッシュボード開発者に確認（5分）

**質問**:
```
Q1: 15スタンプQRコードのペイロードを見せてください
    期待値: {"type":"premium","stamps":15}
    実際の値: ?

Q2: QRコードを複数回生成したことはありますか？
    - 最初の生成: いつ？
    - 再生成: したか？いつ？

Q3: テスト用QRコードを生成したことはありますか？
    - ある → そのペイロードは？
    - ない → OK
```

### 2. 院内のQRコード現物確認（10分）

**手順**:
1. 院内の全てのQRコード設置場所を確認
   - 受付
   - 待合室
   - 診察室
   - その他

2. スマホでそれぞれのQRコードをスキャン
   - LIFF アプリではなく、**スマホの標準カメラ**でスキャン
   - QRコードの中身（文字列）を直接確認できる
   - それぞれのペイロードを記録

3. 全て同じペイロードか確認
   - 全て同じ → 候補2（LIFF の誤読）の可能性
   - 違うペイロードが混在 → 🎉 原因特定！

### 3. イベントログで確認（5分）

**Supabase で以下のSQLを実行**:
```sql
-- QR生値を確認
SELECT
  created_at,
  metadata->>'qr_raw_value' as qr_raw,
  metadata->'qr_parsed'->>'type' as type,
  metadata->'qr_parsed'->>'stamps' as stamps,
  metadata->>'device_type' as device
FROM event_logs
WHERE event_name = 'stamp_scan_api_request'
  AND created_at >= current_date - interval '7 days'
ORDER BY created_at DESC
LIMIT 30;
```

**確認ポイント**:
- `qr_raw` に `"stamps":10` と `"stamps":15` の両方が混在していないか
- 混在している → 🎉 原因特定（複数のQRコードが存在）
- 全て `"stamps":10` → QRコード生成時のミス
- 全て `"stamps":15` → LIFF の誤読の可能性
