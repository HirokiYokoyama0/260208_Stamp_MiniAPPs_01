## Supabase連携機能

| 機能 | 説明 |
|-----|------|
| **ユーザー情報保存** | LINEログイン時に自動的にprofilesテーブルにUPSERT（id, line_user_id, display_name, picture_url） |
| **スタンプ数表示** | Supabaseから `stamp_count` を取得してリアルタイム表示（Single Source of Truth） |
| **スタンプ履歴管理** | stamp_historyテーブルで来院履歴を記録、トリガーでprofilesを自動更新 |
| **重複チェック** | 同日同QRの二重登録を防止 |
| **会員証番号表示** | Supabaseから `ticket_number` を取得して表示（未登録時は「未登録」） |
| **最終アクセス日時** | `updated_at` を日本語フォーマット（例: 2026年2月8日 20:45）で表示 |
| **特典交換システム** | rewardsテーブルで特典マスター管理、reward_exchangesテーブルで交換履歴記録 |
| **積み上げ式スタンプ** | 特典交換後もスタンプ数は減らない、条件を満たせば何度でも交換可能 |
| **友だち登録フラグ** | `liff.getFriendship()` の結果を profiles.is_line_friend に保存 |
| **表示モード保存** | profiles.view_mode で大人用/子供用モードをユーザーごとに永続化 |
| **家族機能** | families/profilesテーブル、招待コード、家族合算スタンプ、代理管理メンバー |
| **アンケート機能** | surveys/survey_questions/survey_responsesテーブル、回答報酬スタンプ付与 |
| **スロットゲーム** | ランダムスタンプ付与（3〜8個）、子供モード専用ボタン |

## アーキテクチャ: データアクセスパターン（ハイブリッド型）

**最終更新:** 2026-03-07

このプロジェクトでは、**読み取り専用操作**と**書き込み操作**で異なるデータアクセスパターンを採用しています。

### 基本方針

| 操作タイプ | アクセス方法 | 理由 |
|-----------|-------------|------|
| **読み取り（SELECT）** | クライアント側から直接Supabase | レスポンス速度、サーバー負荷削減 |
| **書き込み（INSERT/UPDATE/DELETE）** | Next.js API経由 | ビジネスロジック集約、監査ログ取得 |

### 📖 読み取りパターン（クライアント側から直接Supabase）

**使用場面:**
- ユーザー情報取得（プロフィール、家族情報）
- スタンプ数・履歴の表示
- 特典一覧の取得
- 治療記録の閲覧

**実装例:**
```typescript
// components/(adult)/AdultHome.tsx
import { supabase } from '@/lib/supabase';

const { data, error } = await supabase
  .from("profiles")
  .select("stamp_count, updated_at, ticket_number, family_id, real_name")
  .eq("id", userId)
  .single();
```

**RPC関数の直接呼び出し例:**
```typescript
// ケア記録機能（患者向け表示）
const { data, error } = await supabase.rpc('get_latest_dental_record', {
  p_patient_id: userId
});
```

**セキュリティ:**
- `SUPABASE_ANON_KEY` を使用（公開キー）
- RLS（Row Level Security）ポリシーで適切にアクセス制御
- 読み取り専用なのでデータ改ざんリスクなし

### ✍️ 書き込みパターン（Next.js API経由）

**使用場面:**
- スタンプ登録（QRスキャン、手動追加）
- 特典交換
- 家族作成・参加・更新
- アンケート回答送信
- プロフィール更新

**実装例:**
```typescript
// components/(adult)/AdultHome.tsx
const response = await fetch("/api/stamps/manual", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    userId: profile.userId,
    stamps: adjustedStamps,
    staffPin: pin,
  }),
});
```

**Next.js API側の実装例:**
```typescript
// app/api/stamps/manual/route.ts
import { supabase } from "@/lib/supabase";

export async function POST(request: Request) {
  // バリデーション
  // ビジネスロジック（重複チェック、制限チェック等）
  // Supabaseへの書き込み
  const { data, error } = await supabase
    .from("stamp_history")
    .insert({...});

  // 監査ログ記録（イベントログ等）
  await supabase.from("event_logs").insert({...});

  return Response.json({ success: true });
}
```

**セキュリティ:**
- サーバー側で認証・認可チェック
- ビジネスロジックをクライアント側から隠蔽
- トランザクション制御が可能
- 監査ログを確実に記録

### 🔄 ハイブリッド型を採用する理由

**メリット:**
1. **パフォーマンス最適化** - 読み取りはクライアント側で高速化
2. **セキュリティ強化** - 書き込みはサーバー側で厳重管理
3. **開発効率** - 単純な読み取りはAPI不要でコード削減

**注意点:**
- 開発者は「読み取り」か「書き込み」かを意識してパターンを選択
- RLSポリシーの適切な設定が必須（読み取り専用でも漏洩リスクあり）

### 📋 機能別パターン一覧

| 機能 | 読み取り | 書き込み |
|-----|---------|---------|
| **ユーザー情報** | 直接Supabase（profiles SELECT） | API経由（/api/users/me, /api/profiles/[profileId]） |
| **スタンプ** | 直接Supabase（stamp_history SELECT） | API経由（/api/stamps, /api/stamps/manual, /api/stamps/scan） |
| **特典** | 直接Supabase（rewards SELECT） | API経由（/api/rewards/exchange） |
| **家族** | 直接Supabase（families/family_members SELECT） | API経由（/api/families/*） |
| **ケア記録** | 直接Supabase RPC（get_latest_dental_record等） | - （患者向けは読み取りのみ） |
| **アンケート** | 直接Supabase（surveys SELECT） | API経由（/api/survey/submit） |

### ⚠️ 将来の変更可能性

**現状:** ハイブリッド型（読み取り=直接、書き込み=API経由）

**将来検討事項:**
- セキュリティ要件の厳格化により全てAPI経由に統一する可能性あり
- パフォーマンス要件により読み取りパターンを拡大する可能性あり

**変更時の影響範囲:**
- 直接Supabaseアクセスしている全コンポーネント
- 該当: `components/(adult)/AdultHome.tsx`、ケア記録機能等
