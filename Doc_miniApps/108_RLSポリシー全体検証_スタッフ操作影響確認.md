## 🔍 詳細な操作内容

### 1. スタンプ数変更 (`/api/stamps/manual`)

**仕様書**: [archive/Staff_Manual_Stamp_Specification.md](archive/Staff_Manual_Stamp_Specification.md)

**使用ケース**:
- QR読み取り失敗時に手動でスタンプ追加（5 → 6）
- 誤登録時にスタンプ減算（6 → 5）
- データ不整合時に直接設定（5 → 10）

**データベース操作**:

```typescript
// app/api/stamps/manual/route.ts

// ❌ UPDATE は使わない
// DELETE: 全履歴を削除
const { error: deleteError } = await supabase
  .from("stamp_history")
  .delete()
  .eq("user_id", userId);

// INSERT: 新しい起点レコードを追加
const { error: insertError } = await supabase.from("stamp_history").insert({
  user_id: userId,
  stamp_number: newStampCount,  // 新しいスタンプ数
  stamp_method: "manual_admin",
  amount: changeAmount,  // 変更量（+5, -10など）
  notes: `スタッフ操作: ${changeAmount}個 (${currentStampCount} → ${newStampCount})`
});

// UPDATE: profiles.stamp_count を直接更新
const { error: updateError } = await supabase
  .from("profiles")  // ← profiles テーブルへのUPDATE
  .update({ stamp_count: newStampCount })
  .eq("id", userId);
```

**使用する操作**:
- ✅ `stamp_history` DELETE (全履歴)
- ✅ `stamp_history` INSERT (新レコード)
- ✅ `profiles` UPDATE (stamp_count)
- ❌ `stamp_history` UPDATE **使用しない**

## ⚠️ 発見された問題

### 問題1: ポリシー名の不一致

**016B**:
```sql
DROP POLICY IF EXISTS "allow_public_delete" ON stamp_history;
```

**026B**:
```sql
-- "allow_public_delete" ではなく "stamp_history_delete_policy" を作成
CREATE POLICY "stamp_history_delete_policy" ...
```

**027**:
```sql
-- "stamp_history_delete_policy" を DROP していない
DROP POLICY IF EXISTS "allow_public_delete" ON stamp_history;
```

**結果**: 026Bが適用されている場合、027のDROP文は何もしない → 2つのDELETEポリシーが共存

---

### 問題2: UPDATEポリシーの扱い

**現状**:
- 016B: `allow_public_update` (USING true) - UPDATE許可
- 026B: `stamp_history_deny_update` (USING false) - UPDATE禁止

**しかし**: `stamp_history` への UPDATE操作は**一切使われていない**

**結論**: **UPDATEポリシーは無害**（どちらでも影響なし）

---

### 問題3: DELETEポリシーの重複

**シナリオ1**: 016Bのみ適用済み
- `allow_public_delete` (USING true) が有効
- ✅ スタッフ操作: 動作する

**シナリオ2**: 016B + 026B 適用済み
- `allow_public_delete` (USING true) - 削除
- `stamp_history_delete_policy` (UUID形式チェック) - 新規作成
- ❌ スタッフ操作: **UUID形式チェックで一部レコードが削除できない可能性**

**シナリオ3**: 016B + 026B + 027 適用
- `allow_public_delete` (USING true) - 削除済み（027で）
- `stamp_history_delete_policy` (UUID形式チェック) - 残存（027でDROPされていない）
- `allow_delete_all_stamps` (USING true) - 新規作成（027で）
- **PostgreSQLのポリシー評価**: OR条件で評価されるため、どれか1つが true なら許可
- ✅ スタッフ操作: **動作する** （allow_delete_all_stamps が true なので）

---

## 🔧 正しい修正方針

### オプションA: 027を修正（推奨）

**理由**:
- 026Bとの整合性を保つ
- 既存のポリシーを全てDROPしてから新規作成

```sql
-- 027_allow_delete_non_admin_stamps.sql (修正版)

-- 既存のDELETEポリシーを全て削除
DROP POLICY IF EXISTS "allow_public_delete" ON stamp_history;
DROP POLICY IF EXISTS "stamp_history_delete_policy" ON stamp_history;  -- ← 追加

-- 新しいDELETEポリシー: 全レコード削除可能
CREATE POLICY "allow_delete_all_stamps"
  ON stamp_history
  FOR DELETE
  USING (true);
```

---

### オプションB: 026Bを修正

**理由**:
- 027をシンプルに保つ

```sql
-- 026B_minimal_rls_hardening_fixed.sql (修正版)

-- stamp_history の既存ポリシーを削除
DROP POLICY IF EXISTS "allow_public_delete" ON stamp_history;
DROP POLICY IF EXISTS "allow_public_update" ON stamp_history;

-- 新規作成
CREATE POLICY "stamp_history_delete_policy" ...
CREATE POLICY "stamp_history_deny_update" ...
```

## 🎯 結論

### UPDATEポリシーについて

**回答**: **問題ありません**
- `stamp_history` テーブルへのUPDATE操作は一切使われていません
- 026Bで `stamp_history_deny_update` (USING false) を作成しても影響なし
- 実際に使われているのは:
  - ✅ `stamp_history` DELETE
  - ✅ `stamp_history` INSERT
  - ✅ `profiles` UPDATE

### DELETEポリシーについて

**回答**: **修正が必要**
- 027で`stamp_history_delete_policy`もDROPする必要がある
- そうしないと、026Bと027のポリシーが重複する

### 全体的な影響

**回答**: **修正すれば影響なし**
- スタッフ操作の全機能（スタンプ数変更、QR削除）は正常に動作します
- 027を修正してから適用すれば問題ありません
