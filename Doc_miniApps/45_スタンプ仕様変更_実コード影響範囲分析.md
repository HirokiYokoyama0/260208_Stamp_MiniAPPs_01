| 患者区分 | スタンプ数 | 1日制限 | stamp_method |

| **購買インセンティブ（Purchase）** | **5個** | **制限なし** | `"purchase_incentive"` ← **新規** |

### 1. スタンプ数のハードコード箇所

#### ✅ 実際にスタンプ数が記載されている場所

| ファイル | 行番号 | 現在の値 | 内容 |
|---------|-------|---------|------|
| [app/api/stamps/scan/route.ts](app/api/stamps/scan/route.ts) | 13-14 | `10`, `5` | **コメントのみ** |
| [app/api/stamps/auto/route.ts](app/api/stamps/auto/route.ts) | 15, 38 | `5`, `10` | **バリデーション（実処理）** |
| [app/auto-stamp/page.tsx](app/auto-stamp/page.tsx) | 48 | `5`, `10` | **バリデーション（実処理）** |
| [app/scan/page.tsx](app/scan/page.tsx) | 15-16, 261, 273, 279 | `10`, `5` | **コメント + テスト用UI** |

**重要な発見：**
- **スタンプ数は実際にはクライアント側（QRコードペイロード）から送信される**
- **サーバー側のバリデーションは `[5, 10]` の範囲チェックのみ**

#### 📊 現在の実装フロー

```
QRコード（ペイロード or URL）
  ↓ {"type":"premium","stamps":10} または ?amount=10
クライアント側（app/scan/page.tsx or app/auto-stamp/page.tsx）
  ↓ ペイロードのstamps値をそのまま送信
API（/api/stamps/scan or /api/stamps/auto）
  ↓ バリデーション: stamps が 5 または 10 か？
  ↓ OK → stamp_history に amount: stamps を記録
データベース
```

**つまり：**
- スタンプ数はQRコードに埋め込まれている
- APIは `5` または `10` のみを受け付ける
- **この制限を変更する必要がある**

## ⚠️ 注意事項と検討事項

### 1. locationパラメータの使い方

**現在のカメラ用QR URL形式：**
```
?action=stamp&type=qr&amount=10&location=premium
?action=stamp&type=qr&amount=5&location=standard
```

**購買インセンティブを追加する場合：**
```
?action=stamp&type=qr&amount=5&location=purchase
```

**問題点：**
- `type=qr` は共通なので、`location` で区別する必要がある
- または、新しい `type=purchase` を追加する

**推奨：locationで判定**
```typescript
// app/api/stamps/auto/route.ts
if (location === "purchase") {
  // 購買インセンティブ
  // - stamp_method: "purchase_incentive"
  // - 1日制限なし
} else {
  // 来院QR（premium/standard）
  // - stamp_method: "qr"
  // - 1日1回制限
}
```

### 2. 購買インセンティブの「制限なし」のリスク

**問題：**
- 悪意のあるユーザーが何度もスキャンできてしまう
- 不正利用の可能性

**対策案：**

#### 案A：1日あたりの上限を設定
```typescript
// 購買インセンティブも1日3回まで、など
const MAX_PURCHASE_PER_DAY = 3;

const { data: todayPurchases } = await supabase
  .from("stamp_history")
  .select("id")
  .eq("user_id", userId)
  .eq("stamp_method", "purchase_incentive")
  .gte("visit_date", `${today}T00:00:00`)
  .lt("visit_date", `${today}T23:59:59.999Z`);

if (todayPurchases && todayPurchases.length >= MAX_PURCHASE_PER_DAY) {
  return NextResponse.json({
    success: false,
    message: "本日の購買インセンティブの上限に達しました",
  }, { status: 429 });
}
```

#### 案B：スタッフ承認フロー
- QRスキャン後、スタッフが承認ボタンを押す
- 実装が複雑

#### 案C：動的QR生成
- 購買時に1回限り有効なQRを生成
- 実装が複雑

**推奨：案A（1日3回まで）**

---

### 3. 後方互換性

**旧QRコード（Premium: 10, Regular: 5）がまだ流通している可能性**

#### 対策案A：後方互換性を維持（推奨）

```typescript
// APIバリデーション
if (![5, 10, 15].includes(amount)) {
  return NextResponse.json(
    { success: false, message: "無効なスタンプ数です" },
    { status: 400 }
  );
}
```

これにより：
- 旧QRコード（10個）→ 10個付与
- 新QRコード（15個）→ 15個付与
- 両方とも受け付ける

**移行期間後に旧QRコードを回収・廃棄**

#### 対策案B：完全移行

旧QRコードを全て回収後、バリデーションを厳格化：
```typescript
// Premium: 15のみ、Regular: 10のみ受け付ける
if ((type === "premium" && amount !== 15) ||
    (type === "regular" && amount !== 10) ||
    (type === "purchase" && amount !== 5)) {
  return NextResponse.json(
    { success: false, message: "無効なQRコードです" },
    { status: 400 }
  );
}
```

**推奨：案A（後方互換性維持）**
