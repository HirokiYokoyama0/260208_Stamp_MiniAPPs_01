## 📢 問題の概要

**報告された問題:**
> アプリ内スキャン用（ペイロード型）のQRスキャンで、1日1回の制約が効いていない

**現象:**
- premium/regularタイプのQRコードを同じ日に複数回スキャンできてしまう
- カメラ用QRコード（直接LIFF起動）では制約が効いているように見えたが、同じ問題が潜在的に存在

---

## 🔍 根本原因

### タイムゾーンの不一致

両方のAPIで**UTCの日付**を使って「今日」を判定していたため、日本時間（JST）とのズレが発生していました。

#### 問題のあったコード

```typescript
// ❌ 問題のあった実装（UTC基準）
const today = new Date().toISOString().split("T")[0];  // "2026-03-28" (UTC)
const { data: todayQrRecords } = await supabase
  .from("stamp_history")
  .select("id, stamp_method, notes")
  .eq("user_id", userId)
  .eq("stamp_method", "qr")
  .gte("visit_date", `${today}T00:00:00`)
  .lt("visit_date", `${today}T23:59:59.999Z`);
```

#### 問題の詳細

日本時間（JST = UTC+9）では、以下のような問題が発生します：

| 日本時間 (JST) | UTC時間 | `today` の値 | 判定結果 |
|---------------|---------|-------------|---------|
| 2026-03-28 01:00 | 2026-03-27 16:00 | "2026-03-27" | ❌ 前日扱い |
| 2026-03-28 10:00 | 2026-03-28 01:00 | "2026-03-28" | ✅ 当日扱い |
| 2026-03-28 23:00 | 2026-03-28 14:00 | "2026-03-28" | ✅ 当日扱い |
| 2026-03-29 01:00 | 2026-03-28 16:00 | "2026-03-28" | ❌ 前日扱い |

**結果:**
- JST 2026-03-28 01:00にスキャン → UTC 2026-03-27扱い
- JST 2026-03-28 10:00にスキャン → UTC 2026-03-28扱い
- → 同じ日（JST）に2回スキャンしても、UTCでは異なる日付なので制限が効かない

## ✅ 修正内容

### 修正したファイル

1. [app/api/stamps/scan/route.ts](app/api/stamps/scan/route.ts#L111-L123) - アプリ内スキャン用（ペイロード型）
2. [app/api/stamps/auto/route.ts](app/api/stamps/auto/route.ts#L49-L61) - カメラ用QRコード（直接LIFF起動）

### 修正後のコード

```typescript
// ✅ 修正後の実装（JST基準）
// 日本時間（JST = UTC+9）で当日の範囲を計算
const nowJST = new Date(new Date().getTime() + 9 * 60 * 60 * 1000);
const todayJST = nowJST.toISOString().split("T")[0];
const startOfDayUTC = `${todayJST}T00:00:00+09:00`; // JSTの0時をISO形式で
const endOfDayUTC = `${todayJST}T23:59:59.999+09:00`; // JSTの23:59:59をISO形式で

const { data: todayQrRecords } = await supabase
  .from("stamp_history")
  .select("id, stamp_method, notes")
  .eq("user_id", userId)
  .eq("stamp_method", "qr")
  .gte("visit_date", startOfDayUTC)
  .lt("visit_date", endOfDayUTC);
```

### 修正のポイント

1. **日本時間で「今日」を計算**: `new Date().getTime() + 9 * 60 * 60 * 1000`
2. **タイムゾーンオフセット付きISO形式**: `+09:00` を明示
3. **JSTの0時〜23:59:59.999の範囲でクエリ**

## 🎯 修正後の動作

### 正しい判定

| 日本時間 (JST) | 判定範囲 | 結果 |
|---------------|---------|------|
| 2026-03-28 01:00 | 2026-03-28T00:00:00+09:00 〜 23:59:59.999+09:00 | ✅ 当日扱い |
| 2026-03-28 10:00 | 2026-03-28T00:00:00+09:00 〜 23:59:59.999+09:00 | ✅ 当日扱い |
| 2026-03-28 23:00 | 2026-03-28T00:00:00+09:00 〜 23:59:59.999+09:00 | ✅ 当日扱い |
| 2026-03-29 01:00 | 2026-03-29T00:00:00+09:00 〜 23:59:59.999+09:00 | ✅ 翌日扱い |

### 1日1回制限の仕様

| QRタイプ | 1日あたりの制限 | 備考 |
|---------|---------------|------|
| **Premium (15スタンプ)** | ✅ **1回のみ** | 来院時の特典 |
| **Regular (10スタンプ)** | ✅ **1回のみ** | 来院時の特典 |
| **Purchase (5スタンプ)** | ⚪ **制限なし** | 購買インセンティブ |

**重要な仕様:**
- Premium/Regular は **どちらか一方を1回スキャンしたら、その日はもうスキャンできない**
- 例: Premiumを朝にスキャン → その日はRegularもスキャン不可
- 購買インセンティブ（Purchase）のみ何度でもスキャン可能

## 📊 データベースへの影響

### 変更なし

- データベーススキーマの変更は不要
- 既存の `stamp_history` データに影響なし
- `stamp_method = "qr"` の判定ロジックは変更なし

### クエリの違い

#### 修正前のクエリ

```sql
SELECT * FROM stamp_history
WHERE user_id = 'Uxxxxx'
  AND stamp_method = 'qr'
  AND visit_date >= '2026-03-28T00:00:00'
  AND visit_date < '2026-03-28T23:59:59.999Z';
```

#### 修正後のクエリ

```sql
SELECT * FROM stamp_history
WHERE user_id = 'Uxxxxx'
  AND stamp_method = 'qr'
  AND visit_date >= '2026-03-28T00:00:00+09:00'
  AND visit_date < '2026-03-28T23:59:59.999+09:00';
```

**違い**: タイムゾーンオフセット `+09:00` を明示することで、PostgreSQLが正しく日本時間の範囲で検索します。

### 1. タイムゾーン設定の一元管理

現在は各APIで個別にJST変換していますが、将来的には：

```typescript
// lib/timezone.ts（例）
export function getToday JST(): string {
  const nowJST = new Date(new Date().getTime() + 9 * 60 * 60 * 1000);
  return nowJST.toISOString().split("T")[0];
}

export function getJSTRange(dateString: string) {
  return {
    start: `${dateString}T00:00:00+09:00`,
    end: `${dateString}T23:59:59.999+09:00`,
  };
}
```
