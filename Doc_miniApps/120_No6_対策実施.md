**関連ドキュメント**: [120_スタンプ二重付与問題_データ不整合調査レポート.md](./120_スタンプ二重付与問題_データ不整合調査レポート.md)

## 📋 問題の概要

### 報告内容

```
運用担当者からの報告:
垣矢直子さん
- 4/6: 5ポイントQR読み込み → 10ポイントが付与された（5ポイント多い）
- 4/7朝: 10ポイント → 15ポイントに増加（さらに5ポイント増加）
- 4/7: 管理画面から5ポイントに手動修正済み
```

### 調査結果

**データベース検証:**
- `stamp_history`: 2件のみ（1件の正常スキャン + 1件の手動修正）
- `event_logs`: 4件（3件のQRスキャン成功ログ）
- **問題**: 2件のスキャンが `stamp_history` に記録されていない

**イベントログの詳細:**

| 時刻 | current | added | new | stamp_history_id | 実際に存在? |
|------|---------|-------|-----|------------------|-----------|
| 12:15:04 | 0 | 5 | 5 | `b31baa8f...` | ✅ 存在 |
| 12:16:50 | 5 | 5 | 10 | `ee7eea40...` | ❌ **存在しない** |
| 12:16:50 | 10 | 5 | 15 | `fabc203d...` | ❌ **存在しない** |

**重要な発見**: 12:16:50に**同一秒**で2回のスキャン成功が記録されている。

---

## 🔍 根本原因の特定

### 原因1: スマホカメラ/Android OS側の二重送信バグ

**証拠:**
- 垣矢さんの端末: Android 14, Sony Xperia (XQ-CC44)
- 人間が0秒で2回QRコードをスキャンすることは**物理的に不可能**
- カメラアプリまたはAndroid OSがLIFF URLを2回起動した可能性

### 原因2: 並列実行による競合

**発生メカニズム:**

```
12:16:50.000 - リクエスト1
  ├─ stamp_history INSERT 試行 (ee7eea40...)
  │  └─ 何らかの理由で失敗（RLS? デッドロック?）
  ├─ historyError が null（検出漏れ）
  └─ profiles UPDATE 成功 (5 → 10)

12:16:50.xxx - リクエスト2（ほぼ同時）
  ├─ profiles SELECT (stamp_count: 10を取得)
  ├─ stamp_history INSERT 試行 (fabc203d...)
  │  └─ 失敗
  └─ profiles UPDATE 成功 (10 → 15)
```

**結果**: `stamp_history` にレコードがないのに `profiles.stamp_count` だけが増加

### なぜ他のユーザーで問題が出ないのか？

- ✅ 40人のユーザー全員が正常動作している
- ❌ 垣矢さんだけが「同一秒に2回スキャン」という極めて稀な状況に遭遇
- **結論**: 問題はAPIの構造的な欠陥ではなく、**並列実行による競合**

---

## ✅ 実施した対策

### 対策: 5秒以内の重複リクエスト防止

両方のスタンプ付与APIに**重複リクエスト検出ロジック**を追加しました。

#### 修正ファイル

1. **[app/api/stamps/auto/route.ts](../app/api/stamps/auto/route.ts)** (カメラ用QR直接起動)
   - Line 102-149: 5秒以内の重複リクエスト防止チェックを追加

2. **[app/api/stamps/scan/route.ts](../app/api/stamps/scan/route.ts)** (ミニアプリ内スキャン)
   - Line 183-224: 5秒以内の重複リクエスト防止チェックを追加

#### 実装内容

```typescript
// 🔒 5秒以内の重複リクエスト防止チェック
const { data: recentStamps, error: recentCheckError } = await supabase
  .from("stamp_history")
  .select("created_at, amount, stamp_method")
  .eq("user_id", userId)
  .order("created_at", { ascending: false })
  .limit(1)
  .single();

if (!recentCheckError && recentStamps) {
  const lastStampTime = new Date(recentStamps.created_at);
  const now = new Date();
  const diffMs = now.getTime() - lastStampTime.getTime();

  // 5秒以内の連続スキャンを拒否
  if (diffMs < 5000) {
    console.log(`⚠️ [重複リクエスト防止] User ${userId}, 前回から${diffMs}ms, 拒否`);
    console.log(`   前回: ${lastStampTime.toISOString()}, 今回: ${now.toISOString()}`);

    // イベントログ記録（失敗: 重複リクエスト）
    await logStampScanFail({
      error: "Duplicate request within 5 seconds",
      userId: userId,
      errorType: "duplicate_request",
      httpStatus: 429,
      requestType: type,
      requestStamps: amount,
    });

    return NextResponse.json(
      {
        success: false,
        message: "処理中です。少しお待ちください。",
        duplicateRequest: true,
        waitSeconds: Math.ceil((5000 - diffMs) / 1000)
      },
      { status: 429 } // 429 Too Many Requests
    );
  }

  console.log(`✅ [重複リクエストチェック] 前回から${diffMs}ms経過、処理続行`);
}
```

### 🔴 検討したが実施しなかった対策

#### ❌ 対策案1: 手動UPDATEの削除（トリガーのみに任せる）

**なぜ実施しなかったか:**

1. **実績データが反証している**
   - 40人全員が正常動作している
   - もし手動UPDATEとトリガーの競合が構造的な問題なら、全ユーザーで頻繁に問題が起きるはず

2. **トリガーは「必ず実行される」保証がない**
   - RLSポリシーによるブロックの可能性
   - 権限不足の可能性
   - ロック競合の可能性
   - トリガー自体の無効化の可能性

3. **手動UPDATEは二重保険として機能**
   - トリガーが失敗した場合の安全網
   - `/api/stamps/auto` はエラーチェックがある（より安全）

**結論**: 手動UPDATEを削除すると、40人の正常動作を壊すリスクがある。

## 🔬 対策の妥当性検証

### なぜ5秒制限が適切か？

1. **人間の行動パターン**
   - QRコードをスキャン → カメラをかざす → ピント合わせ → 読み取り
   - この一連の動作は最低でも3〜5秒かかる
   - 5秒以内の再スキャンは**機械的な二重送信のみ**

2. **ユーザー体験への影響**
   - 誤って2回スキャンしてしまった場合でも、5秒待てば再スキャン可能
   - エラーメッセージで待ち時間を明示（`waitSeconds`）

3. **実装の単純さ**
   - データベースの最新レコードを1件取得するだけ
   - インデックスが効いているため高速

### 想定されるエッジケース

| ケース | 挙動 | 適切か？ |
|--------|------|---------|
| 同じユーザーが5秒後に再スキャン | ✅ 許可 | ✅ 正常な再訪問 |
| 異なるユーザーが同時にスキャン | ✅ 許可 | ✅ 問題なし |
| ネットワーク遅延で2回送信 | ❌ 2回目拒否 | ✅ 正しい動作 |

## 🚀 今後の監視項目

### 監視すべきメトリクス

1. **429エラーの発生頻度**
   - `event_logs` で `errorType: "duplicate_request"` をカウント
   - 頻繁に発生する場合は、アプリ側のバグを疑う

2. **stamp_history と profiles.stamp_count の整合性**
   - 定期的に `MAX(stamp_number)` と `stamp_count` を比較
   - 不整合があれば調査

3. **5秒制限の妥当性検証**
   - 正常なユーザーが拒否されていないかログで確認
   - 必要に応じて3秒や10秒に調整

### アラート条件

```sql
-- 1時間に10回以上の重複リクエストがあるユーザー
SELECT
  user_id,
  COUNT(*) as duplicate_count
FROM event_logs
WHERE
  event_name = 'stamp_scan_fail'
  AND metadata->>'errorType' = 'duplicate_request'
  AND created_at > NOW() - INTERVAL '1 hour'
GROUP BY user_id
HAVING COUNT(*) > 10;
```
