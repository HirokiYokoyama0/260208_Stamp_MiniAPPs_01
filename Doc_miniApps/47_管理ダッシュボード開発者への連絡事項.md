## 📋 要約

スタンプ仕様変更に伴い、**QRコード生成ページ**の調整が必要です。

---

## 🎯 変更内容

### 現在の仕様

| 患者区分 | スタンプ数 | 1日制限 |
|---------|-----------|--------|
| 優良患者様（Premium） | **10個** | 1回のみ |
| 通常患者様（Regular） | **5個** | 1回のみ |

### 変更後の仕様

| 患者区分 | スタンプ数 | 1日制限 |
|---------|-----------|--------|
| 優良患者様（Premium） | **15個** (+5) | 1回のみ |
| 通常患者様（Regular） | **10個** (+5) | 1回のみ |
| **購買インセンティブ（Purchase）** | **5個** | **制限なし** ← **新規** |

## 🔧 管理ダッシュボード側で必要な変更

### 1. QRコード生成ページの調整

#### 1-1. スタンプ数の変更

**現在：**
```
Premium: amount=10
Regular: amount=5
```

**変更後：**
```
Premium: amount=15
Regular: amount=10
```

#### 1-2. 購買インセンティブQRの追加

**新規追加が必要：**
```
Purchase: amount=5, type=purchase, location=shop
```

**重要：** Doc 48の実装調査により、`location` パラメータは以下を使用します：
- 来院QR: `location=entrance`（実装済み）
- 購買QR: `location=shop`（新規）

---

### 2. QRコード生成の具体的な変更内容

#### カメラ用QRコード（直接LIFF起動型）

**⚠️ 重要：Doc 48の実装調査結果に基づく正しいURL形式**

**現在のURL形式（実装済み）：**
```
優良: https://liff.line.me/{LIFF_ID}?action=stamp&type=qr&amount=10&location=entrance
通常: https://liff.line.me/{LIFF_ID}?action=stamp&type=qr&amount=5&location=entrance
```

**変更後のURL形式：**
```
優良: https://liff.line.me/{LIFF_ID}?action=stamp&type=qr&amount=15&location=entrance
通常: https://liff.line.me/{LIFF_ID}?action=stamp&type=qr&amount=10&location=entrance
購買: https://liff.line.me/{LIFF_ID}?action=stamp&type=purchase&amount=5&location=shop  ← 新規
```

**補足説明：**
- `location=entrance` が実際のデータベースで使用されている値です
- `location=premium` や `location=standard` は使用されていません（Doc 48で確認済み）
- 購買インセンティブは `type=purchase` で区別します

#### アプリ内スキャン用QRコード（ペイロード型）

**現在のペイロード：**
```json
{"type":"premium","stamps":10}
{"type":"regular","stamps":5}
```

**変更後のペイロード：**
```json
{"type":"premium","stamps":15}
{"type":"regular","stamps":10}
{"type":"purchase","stamps":5}  ← 新規追加
```

## ⚠️ 重要な確認事項

### 1. 後方互換性について

**質問：旧QRコード（Premium: 10, Regular: 5）はまだ流通していますか？**

- **はい** → ミニアプリ側は後方互換性を維持します（両方受け付けます）
- **いいえ** → 厳格なバリデーションに変更可能

**推奨：**
移行期間中（3ヶ月程度）は旧QRコードも引き続き使用可能にし、段階的に新QRコードに置き換える。

---

### 2. 購買インセンティブの制限なしリスク

**問題：**
悪意のあるユーザーが1日に何度もスキャンできてしまう可能性があります。

**対策案：**

#### 案A：1日あたりの上限を設定（推奨）

```
購買インセンティブQRは1日3回まで
```

**管理画面での設定項目：**
```
購買インセンティブQRの1日上限回数: [ 3 ] 回
```

**メリット：**
- 不正利用のリスクを低減
- 実装が比較的簡単

#### 案B：スタッフ承認フロー

QRスキャン後、スタッフが管理画面で承認ボタンを押すまで確定しない。

**デメリット：**
- 実装が複雑
- スタッフの手間が増える

#### 案C：動的QR生成

購買時に1回限り有効なQRコードを生成（ワンタイムQR）。

**デメリット：**
- 実装が複雑
- QR生成・管理の仕組みが必要

**推奨：案A（1日3回まで）**

## 📊 データベースへの影響

### stamp_history テーブル

**変更点：**
新しい `stamp_method` の値が追加されます。

| stamp_method | 説明 | 備考 |
|--------------|------|------|
| `"qr"` | QR来院スタンプ（Premium/Regular） | ✅ 現在の実装 |
| `"qr_scan"` | QR来院スタンプ（旧） | ⚠️ 後方互換性（Doc 48参照） |
| `"purchase_incentive"` | 購買インセンティブ | **新規追加** |

**重要：** Doc 48の調査により、データベースには `"qr"` と `"qr_scan"` の両方が存在することが確認されています。
- `"qr"`: 現在の実装（2024年3月以降）
- `"qr_scan"`: 旧実装（後方互換性のため残存）

**クエリ例：**
```sql
-- 購買インセンティブの利用状況を確認
SELECT
  user_id,
  COUNT(*) as purchase_count,
  SUM(amount) as total_stamps
FROM stamp_history
WHERE stamp_method = 'purchase_incentive'
  AND visit_date >= CURRENT_DATE
GROUP BY user_id
ORDER BY purchase_count DESC;
```

**管理ダッシュボード側で必要になる可能性：**
- 購買インセンティブの利用統計表示
- 不正利用の検知・アラート
