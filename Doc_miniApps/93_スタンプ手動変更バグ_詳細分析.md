## 1. 問題の発見経緯

### 症状

```
スタッフが会員証タブでスタンプ数を変更
63個 → 56個 に変更
  ↓
一瞬 56個 と表示される
  ↓
他のタブに移動すると 63個 に戻る
  ↓
ページリロードしても 63個 のまま
```

### ユーザーからの重要な指摘

> 「**以前はそんな事象はでていなかった**ことをしっかり考慮しろよ。なにか重大な問題がでている。supabaseも確認しろ。。安易にコード修正するな」

この指摘が重要だった理由：
- コードのバグではなく、**データベース側の構造的問題**を示唆
- 新しく発生した現象 = **データの蓄積によって表面化した問題**
- React の状態管理やキャッシュの問題ではない

---

## 2. なぜ「途中でダメになった」のか？

### 問題が発生しなかった時期（過去）

**条件**: スタンプ数が常に増える方向だった

```
来院履歴:
1回目: stamp_number = 10
2回目: stamp_number = 20
3回目: stamp_number = 30
...
現在: stamp_number = 50

MAX(stamp_number) = 50 ✅ 正常
```

この時期は、トリガーの `MAX(stamp_number)` が常に**最新の値**と一致していた。

### 問題が発生するようになった時期（最近）

**条件**: スタンプ数を**減らす操作**が行われるようになった

```
来院履歴:
...
2026-03-01: stamp_number = 63 (スロットゲーム)
2026-03-07: stamp_number = 56 (スタッフが手動で減らした)

MAX(stamp_number) = 63 ❌ 最新値(56)と不一致！
```

### なぜ問題が表面化したのか

**トリガーの計算ロジック**:
```sql
-- supabase/002_create_stamp_history_table.sql (Line 89-93)
stamp_count = (
  SELECT COALESCE(MAX(stamp_number), 0)  -- ← 全レコードから最大値を取得
  FROM stamp_history
  WHERE user_id = NEW.user_id
)
```

**問題点**:
1. トリガーは **全期間の履歴** から `MAX(stamp_number)` を計算
2. 過去に `stamp_number = 63` のレコードがあれば、それが最大値
3. 新しく `stamp_number = 56` を INSERT しても、MAX は 63 のまま
4. 結果: `profiles.stamp_count` が 56 → 63 に上書きされる

### データが増えるほど問題が悪化する理由

```
ユーザーA:
- 累積スタンプ数が少ない → 問題なし
- 過去最大値: 20個 → 15個に減らす → 問題なく動作

ユーザーB:
- 累積スタンプ数が多い → 問題発生
- 過去最大値: 63個 → 56個に減らす → 63個に戻る ❌
```

スタンプ数が多いユーザーほど、過去の最大値が大きく、減らす操作が失敗しやすい。

## 3. なぜ以前の実装では問題なかったのか

### Phase 1: 初期実装（問題なし）

**時期**: 2026年1月〜2月初旬
**運用**: スタンプは基本的に増える一方

- 来院ごとに +1 個
- 特典交換しても減らない（積み上げ式）
- スタッフが手動で増やす操作のみ

**データ例**:
```
stamp_history:
1, 2, 3, 4, 5, 6, ..., 50, 51, 52, ...

MAX(stamp_number) = 52 (常に最新値)
profiles.stamp_count = 52 ✅
```

### Phase 2: スロットゲーム導入（問題の種が生まれる）

**時期**: 2026年2月中旬
**変更**: スロットゲームで大量ポイント付与

- 通常来院: +1個
- スロットゲーム: +5〜10個 (当たり時)

**データ例**:
```
stamp_history:
..., 50, 51, 52, 62 (スロット当たり +10個)

MAX(stamp_number) = 62
profiles.stamp_count = 62 ✅ まだ問題なし
```

この時点では、減らす操作がないため問題は顕在化していない。

### Phase 3: スタッフ手動調整が増加（問題が表面化）

**時期**: 2026年3月
**運用**: スタッフがスタンプ数を調整する頻度が増加

**理由**:
- ユーザーからの問い合わせ対応
- 誤ってスタンプが付与されたケースの修正
- スロットゲームでの過剰付与の調整

**データ例**:
```
stamp_history:
..., 62, 63 (スロット), 56 (スタッフが手動で減らした)
                        ↑
                     最新のレコード

MAX(stamp_number) = 63 ❌
profiles.stamp_count = 63 (56ではなく63に戻る)
```

**問題が顕在化**:
- スタンプ数を**減らす**操作が発生
- トリガーが過去の最大値を参照
- 新しい値が上書きされる

## 4. データベーストリガーの設計思想

### Single Source of Truth の原則

`Doc_miniApps/05_Database_Schema.md` より:

> **設計原則: Single Source of Truth**
> - `profiles.stamp_count` がスタンプ数の唯一の真実
> - 手動で更新する必要なし（トリガーが自動計算）
> - スタンプ数 = `MAX(stamp_number)` （`COUNT(*)` ではない）

### なぜ MAX(stamp_number) を使うのか？

**理由1**: `stamp_number` は「その時点での累積スタンプ数」を表す

```
1回目来院: stamp_number = 1
2回目来院: stamp_number = 2
3回目来院: stamp_number = 3
...
10回目来院: stamp_number = 10

現在のスタンプ数 = MAX(stamp_number) = 10
```

**理由2**: スタッフが途中で調整できる

```
スタッフが「5個に設定」を実行
→ stamp_number = 5 の新しいレコードを INSERT
→ MAX(stamp_number) = 5 になる（はずだった）
```

### 設計の盲点

**想定されていなかったケース**: スタンプ数を**減らす**操作

```
現在: stamp_number = 10
スタッフが「5個に設定」
→ stamp_number = 5 の新しいレコードを INSERT
→ MAX(stamp_number) = 10 (変わらない！) ❌
```

トリガーは **全履歴** から MAX を計算するため、過去の大きい値が残っている限り、減らす操作ができない。

## 5. トリガー間の不整合

### INSERT トリガー (002_create_stamp_history_table.sql)

```sql
CREATE OR REPLACE FUNCTION update_profile_stamp_count()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE profiles
  SET
    stamp_count = (
      SELECT COALESCE(MAX(stamp_number), 0)  -- ← MAX(stamp_number)
      FROM stamp_history
      WHERE user_id = NEW.user_id
    ),
    ...
END;
$$ LANGUAGE plpgsql;
```

**計算方法**: `MAX(stamp_number)`

### DELETE トリガー (016_add_delete_policy_stamp_history.sql)

```sql
CREATE OR REPLACE FUNCTION update_profile_on_stamp_delete()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE profiles
  SET
    stamp_count = (
      SELECT COALESCE(SUM(amount), 0)  -- ← SUM(amount)
      FROM stamp_history
      WHERE user_id = OLD.user_id
    ),
    ...
END;
$$ LANGUAGE plpgsql;
```

**計算方法**: `SUM(amount)`

### 問題点

**同じ `profiles.stamp_count` を異なる方法で計算している**:

| トリガー | 計算方法 | 意味 |
|---------|---------|------|
| INSERT | `MAX(stamp_number)` | 最新の累積スタンプ数 |
| DELETE | `SUM(amount)` | 全履歴の合計ポイント |

**結果**:
- INSERT 後と DELETE 後で `stamp_count` の値が変わる可能性がある
- データベースの整合性が保証されない

## 6. なぜ修正が必要だったのか

### 誤った仮説と修正の試み

#### 仮説1: fetchUserData が古いデータを取得している（❌ 誤り）

**修正**: fetchUserData の呼び出しを削除
**結果**: 効果なし

**理由**: データベース自体が間違った値を返しているため、何度取得しても同じ

#### 仮説2: Supabase Read Replica の遅延（❌ 誤り）

**修正**: API レスポンスの値を直接使用
**結果**: 効果なし

**理由**: レプリカ遅延ではなく、トリガーが即座に上書きしている

#### 仮説3: React の useEffect 再実行問題（❌ 誤り）

**修正**: 依存配列を変更
**結果**: 効果なし

**理由**: React の問題ではなく、データベーストリガーの問題

### 正しい原因の特定

**調査**: データベースを直接確認

```bash
node scripts/check-current-stamp-value.mjs
```

**結果**:
```
profiles.stamp_count: 63
stamp_history 最新レコード: stamp_number = 56

❌ 不一致！
```

**結論**: データベーストリガーが問題

## 7. 正しい修正方法

### 解決策: 過去のレコードを削除してから INSERT

```typescript
// スタンプ数を減らす場合、新しい値より大きいレコードを削除
if (newStampCount < currentStampCount) {
  await supabaseAdmin
    .from("stamp_history")
    .delete()
    .eq("user_id", userId)
    .gt("stamp_number", newStampCount);  // ← 新しい値より大きいレコードを削除
}

// 新しいレコードを INSERT
await supabase.from("stamp_history").insert({
  user_id: userId,
  stamp_number: newStampCount,  // ← これが新しい MAX になる
  ...
});
```

### なぜこの修正で動作するのか

**修正前**:
```
stamp_history:
..., 56, 58, 63 (過去の最大値)

新しく 56 を INSERT
→ ..., 56, 58, 63, 56

MAX(stamp_number) = 63 ❌
```

**修正後**:
```
stamp_history:
..., 56, 58, 63

DELETE WHERE stamp_number > 56
→ ..., 56 (58, 63 が削除される)

新しく 56 を INSERT
→ ..., 56, 56

MAX(stamp_number) = 56 ✅
```

### 念のための手動更新

```typescript
// トリガーが発火するが、DELETE トリガーは SUM(amount) で計算するため
// 念のため手動でも更新する
await supabaseAdmin
  .from("profiles")
  .update({
    stamp_count: newStampCount,
    updated_at: new Date().toISOString()
  })
  .eq("id", userId);
```

**理由**:
- INSERT トリガー: `MAX(stamp_number)` で計算
- DELETE トリガー: `SUM(amount)` で計算
- トリガー間の不整合を回避するため、手動更新で確実性を担保

## 8. RLS (Row Level Security) の制約

### ANON_KEY では DELETE できない

**テスト結果**:
```bash
node scripts/test-delete-permission.mjs

ANON_KEY: 削除件数 = 0 ❌
SERVICE_ROLE_KEY: 削除件数 = 12 ✅
```

**理由**: RLS ポリシーで DELETE 操作が制限されている

### RLS ポリシーの確認

`supabase/016_add_delete_policy_stamp_history.sql`:

```sql
CREATE POLICY "allow_public_delete"
  ON stamp_history
  FOR DELETE
  USING (true);  -- ← 全員削除可能のはずだが...
```

**実際**: ANON_KEY では削除できない（RLS の仕様）

**解決策**: SERVICE_ROLE_KEY を使用

```typescript
const supabaseAdmin = createClient(supabaseUrl, serviceRoleKey);

await supabaseAdmin
  .from("stamp_history")
  .delete()
  .eq("user_id", userId)
  .gt("stamp_number", newStampCount);
```

## 9. データの監査証跡への影響

### 削除されるデータ

**修正により削除されるレコード**:
```
スタンプ数を 63個 → 56個 に減らす場合

削除されるレコード:
- stamp_number = 63 (スロットゲーム)
- stamp_number = 62 (来院)
- stamp_number = 59 (来院)
- stamp_number = 58 (来院)
- stamp_number = 57 (来院)

残るレコード:
- stamp_number = 56 以下の全レコード
```

### 監査証跡の損失

**問題**:
- 過去のスタンプ履歴が**物理削除**される
- 削除されたレコードは復元できない
- 監査証跡として不完全

### 代替案（将来の改善案）

#### 案1: 論理削除

```sql
ALTER TABLE stamp_history
  ADD COLUMN deleted_at TIMESTAMPTZ;

-- 削除時は deleted_at を設定
UPDATE stamp_history
SET deleted_at = NOW()
WHERE user_id = ? AND stamp_number > ?;

-- トリガーは deleted_at IS NULL のレコードのみ計算
```

#### 案2: is_active フラグ

```sql
ALTER TABLE stamp_history
  ADD COLUMN is_active BOOLEAN DEFAULT TRUE;

-- 削除時は is_active = FALSE
UPDATE stamp_history
SET is_active = FALSE
WHERE user_id = ? AND stamp_number > ?;
```

#### 案3: 削除ログテーブル

```sql
CREATE TABLE stamp_history_deleted (
  id UUID PRIMARY KEY,
  original_id UUID,
  user_id TEXT,
  stamp_number INTEGER,
  deleted_at TIMESTAMPTZ,
  deleted_reason TEXT,
  ...
);

-- 削除前に deleted テーブルにコピー
INSERT INTO stamp_history_deleted
SELECT *, NOW(), '手動調整'
FROM stamp_history
WHERE user_id = ? AND stamp_number > ?;

-- その後削除
DELETE FROM stamp_history
WHERE user_id = ? AND stamp_number > ?;
```

## 10. まとめ

### 問題の本質

1. **トリガーの設計**: `MAX(stamp_number)` を全履歴から計算
2. **運用の変化**: スタンプ数を減らす操作が発生
3. **データの蓄積**: 過去の大きい値が残っている
4. **結果**: 新しい小さい値が上書きされる

### なぜ途中でダメになったのか

- **初期**: スタンプ数が増える一方 → 問題なし
- **スロット導入**: 大量付与されるケースが増加 → 問題の種
- **手動調整増加**: 減らす操作が発生 → 問題が表面化

### 修正のポイント

1. **データベース側の問題** → コード修正だけでは解決しない
2. **過去のレコードを削除** → MAX が正しく計算されるようにする
3. **SERVICE_ROLE_KEY を使用** → RLS をバイパスして削除
4. **手動更新で確実性を担保** → トリガー間の不整合を回避

### 学んだこと

1. **ユーザーの指摘を重視**: 「以前は問題なかった」→ データベース調査
2. **データベースを直接確認**: コード修正より先にデータを見る
3. **トリガーの動作を理解**: 暗黙的な処理が予期しない結果を生む
4. **Single Source of Truth**: 設計原則と実装の乖離を見つける
