### ステップ3：ANON_KEYを取得する

**重要：** Supabase ANON_KEYは以下の3つの方法で簡単に取得できます。

#### 方法1：Networkタブから取得（最も簡単）

1. DevToolsの「Network」タブを開く
2. アプリ内で何か操作する（スタンプページを開くなど）
3. Supabaseへのリクエスト（`supabase.co`を含むもの）をクリック
4. Request Headersを見る：

```
apikey: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

この`apikey`の値が **ANON_KEY** です。

#### 方法2：Sourcesタブから取得

1. DevToolsの「Sources」タブを開く
2. 検索機能（Ctrl+Shift+F / Cmd+Shift+F）を使う
3. `NEXT_PUBLIC_SUPABASE_ANON_KEY` または `createClient` で検索
4. JavaScriptファイル内に直接埋め込まれている値が見つかる

例：
```javascript
// lib/supabase.ts がビルド後のJSファイルに変換される
const supabaseAnonKey = "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...";
```

#### 方法3：Consoleから直接取得

Supabaseクライアントがグローバルスコープに露出している場合：

```javascript
// window オブジェクトを検索
Object.keys(window).filter(k => k.includes('supabase'));

// または、既存のリクエストを再利用
performance.getEntriesByType('resource')
  .filter(r => r.name.includes('supabase'))
  .map(r => r.name);
```

## 🔑 ANON_KEYが露出する理由

### Next.jsの仕組み

`NEXT_PUBLIC_`で始まる環境変数は、**クライアント側（ブラウザ）にバンドルされます**。

#### 現在のコード（lib/supabase.ts:4）

```typescript
const supabaseAnonKey = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY ?? "";
```

この値は、ビルド時に以下のように変換されます：

```javascript
// ビルド後のJavaScriptファイル
const supabaseAnonKey = "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...";
```

### Vercelの環境変数でも同じ

Vercelの環境変数に`NEXT_PUBLIC_SUPABASE_ANON_KEY`を設定しても、**ビルド時にJavaScriptファイルに埋め込まれます**。

```
Vercel環境変数
  ↓
Next.jsビルド
  ↓
JavaScriptファイルに直接埋め込み
  ↓
ブラウザでダウンロード（誰でもアクセス可能）
```

### これは仕様

**これはNext.jsの正常な動作です**。`NEXT_PUBLIC_`で始まる環境変数は、クライアント側で使用するために公開されることが前提です。

そのため、**Supabase ANON_KEYは本来「公開されても問題ない」という前提で設計されています**。

### 問題の本質

ANON_KEYが公開されることは問題ではありません。**問題は、ANON_KEYだけで他人のデータにアクセスできてしまうRLSポリシーの設定です**。

#### 正しい設計：
```
ANON_KEY（公開OK）
  ↓
Supabase RLS（アクセス制御）
  ↓
自分のデータのみアクセス可能
```

#### 現在の設計（問題）：
```
ANON_KEY（公開OK）
  ↓
Supabase RLS（制限なし）
  ↓
全員のデータにアクセス可能 🔴
```

## 🎯 結論

### 「他人のデータ閲覧」は極めて容易

1. **必要なスキル：** 中学生レベルのJavaScript知識
2. **必要な時間：** 5分以内
3. **必要なツール：** ブラウザのDevTools（すべてのブラウザに標準搭載）
4. **実行難易度：** 極めて低い

### 最小限の変更では完全に防げない

- ✅ DELETE制限：他人のスタンプ削除を防ぐ
- ✅ 個人情報更新制限：本名・診察券番号の改ざんを防ぐ
- ❌ **閲覧は引き続き可能**：個人情報の漏洩リスクは残る

### 完全に防ぐにはRPC関数が必要

- RPC関数経由でのみアクセス許可
- サーバー側でユーザーIDを検証
- 直接テーブルアクセスは禁止
