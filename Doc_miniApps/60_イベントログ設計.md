## 1. 背景・目的

### 実現したいこと
- **LINE配信の効果測定**: 配信後に何人がアプリを開いたか（`app_open` + `source`）
- **ユーザー行動分析**: どの機能が使われているか、離脱ポイントはどこか
- **不具合調査**: QRスキャンが付与されなかった等の原因追跡

### Supabaseのみで完結する理由
1. リアルタイム性（Table Editor / リアルタイム購読で即確認）
2. JSONB型の柔軟性（後からmetadata項目を追加可能）
3. 集計もSQLで完結
4. 外部サービス不要でコスト効率が良い

実ファイル: [`supabase/015_create_event_logs_table_ForUser.sql`](../supabase/015_create_event_logs_table_ForUser.sql)

```sql
create table if not exists event_logs (
  id uuid default gen_random_uuid() primary key,
  user_id text references profiles(id) on delete cascade,  -- 誰が（LINE User ID / NULL可）
  event_name text not null,                                -- 何を
  source text,                                             -- どこから（流入元）

```

**インデックス（実装済み）**:
- `idx_event_logs_user_id` (user_id)

- `idx_event_logs_user_event` (user_id, event_name) ← 複合
- `idx_event_logs_event_created` (event_name, created_at desc) ← 複合

**集計ビュー（実装済み）**:
- `daily_active_users`：`app_open` の日別DISTINCT user_id（過去400日）

## 3. RLSポリシー（実装準拠：031 / 032 が現行）

現行の実態は [`031_fix_event_logs_rls.sql`](../supabase/031_fix_event_logs_rls.sql) と
[`032_fix_event_logs_insert_policy.sql`](../supabase/032_fix_event_logs_insert_policy.sql)。

| 操作 | anon / authenticated | 管理ダッシュボード |
|------|----------------------|--------------------|
| **SELECT** | ❌ 拒否（`USING (false)`） | ✅ 可（SERVICE_ROLE_KEYでRLSバイパス） |
| **INSERT** | ✅ 可（下記 形式チェック付き） | ✅ 可 |
| **UPDATE** | ❌ 拒否（ログは不変） | （service roleは可） |
| **DELETE** | ❌ 拒否 | ✅ 可（service role） |

**INSERTの WITH CHECK 条件（032）**:
```sql
user_id IS NULL OR                      -- 匿名イベント（userId未指定）
user_id ~ '^U[0-9a-f]{32}$' OR          -- 本番 LINE User ID
user_id ~ '^U_test_' OR                 -- テスト用
user_id LIKE 'manual-child-%'           -- 代理管理の子供アカウント
```

→ **新イベントを追加してもRLS変更は不要**。上記形式の `user_id`（または NULL）であれば anon キーでINSERTが通る。

**RLSの変遷（履歴）**:
- `015`：INSERT `with check(true)` / SELECT `using(true)`（＝anonでも全件SELECT可能な穴があった）
- `026`（minimal_rls_hardening）：INSERTごと禁止してしまい、ログ送信が一時停止
- `031`：SELECT拒否・INSERT形式チェック・UPDATE/DELETE拒否に修正
- `032`：INSERTの WITH CHECK を NULL 優先に修正（031のINSERT不発を解消）

## 4. logEvent 実仕様（lib/analytics.ts 準拠）

実ファイル: [`lib/analytics.ts`](../lib/analytics.ts)

### 4.1 シグネチャ

```typescript
logEvent({
  eventName: string,                 // 必須
  source?: string,                   // 省略時はURLの ?from / ?source、無ければ 'direct'
  metadata?: Record<string, any>,    // 任意の追加情報
  userId?: string,                   // ★明示的に渡すこと（省略すると user_id = null で記録）
})
```

> ⚠️ **userId は自動検出されない**。各コンポーネントで `useLiff()` の `profile.userId` を取得して渡す。
> 渡し忘れると `user_id: null` の匿名ログになり、ユーザー紐付けができない。

### 4.2 自動付与される metadata（クライアント実行時）

`logEvent` 実行時、`window` がある環境では以下が自動でマージされる:
```
user_agent, screen_size (例 375x812), viewport_size, referrer, page_path, timestamp
```
- `source` は `?from` / `?source` から自動検出（無ければ `'direct'`）。

> ⚠️ **既知の盲点**：`page_path` は `window.location.pathname` のみで、**クエリ文字列（`?action=...`）は記録されない**。カメラQRの付与パラメータ追跡には別途metadataで明示的に渡す必要がある（§7参照）。

### 4.3 失敗時の挙動
- INSERT失敗・例外は `console.error` のみで握り潰し、`{ success:false, error }` を返す。
- **ユーザー体験は妨げない**設計。ただし多くの呼び出し元が `await` しているため、クリティカルパスに置く場合はレイテンシに注意。

## 8. 分析用SQL（動作可否を明記）

### 8.1 日別アクティブユーザー数（DAU）✅ 動作する
```sql
select * from daily_active_users
where date >= current_date - interval '7 days'
order by date desc;
```

### 8.2 イベント別集計（人気機能ランキング）✅ 動作する
```sql
select * from event_summary;  -- 直近30日
```

### 8.3 特定ユーザーの行動履歴（時系列）✅ 動作する
```sql
select created_at, event_name, source, metadata
from event_logs
where user_id = 'Uxxxxxxxx...'
  and created_at >= current_date - interval '30 days'
order by created_at asc;
```

### 8.4 スキャン方法別の集計 ✅ 動作する
```sql
select event_name, metadata->>'scan_method' as scan_method,
       count(*) as cnt, avg((metadata->>'stamps_added')::int) as avg_stamps
from event_logs
where event_name in ('stamp_scan_success', 'stamp_scan_success_in_app')
  and created_at >= current_date - interval '7 days'
group by 1, 2 order by 1, 2;
```

### 8.5 スキャン成功率 ⚠️ 部分的
```sql
-- 注意: stamp_scan_start はアプリ内スキャンのみ記録。
--       カメラQRには start イベントが無いため、母数として不完全。
select
  sum((event_name='stamp_scan_start')::int)   as starts_in_app,
  sum((event_name like 'stamp_scan_success%')::int) as successes,
  sum((event_name='stamp_scan_fail')::int)    as failures
from event_logs
where created_at >= current_date - interval '30 days';
```

### 8.6 LINE配信開封率 ⚠️ 現状データなしで実質不可
> `message_delivery_logs`（§2.2）は実DBに存在するが**空（配信記録0件）**のため、開封率は算出できない。
> 配信記録の投入が始まれば機能する。代替として `app_open` の `source` 別集計は可能:
```sql
select coalesce(source,'direct') as source,
       count(*) as opens, count(distinct user_id) as users
from event_logs
where event_name = 'app_open'
  and created_at >= current_date - interval '30 days'
group by 1 order by opens desc;
```

## 10. プライバシー・セキュリティ

- `event_logs` には **user_id のみ**記録。氏名・会員証番号は入れない（分析時に `profiles` とJOIN）。
- 閲覧は **管理ダッシュボード（SERVICE_ROLE_KEY）のみ**。anonからのSELECTはRLSで拒否（§3）。
- INSERTは形式チェック付きで anon 許可（§3）。

## 11. 管理ダッシュボード開発者向け情報

### 11.1 データアクセス
全ログ閲覧は **SERVICE_ROLE_KEY**（RLSバイパス）を使用。
```typescript
import { getSupabaseAdmin } from '@/lib/supabase';
const admin = getSupabaseAdmin();
const { data } = await admin
  .from('event_logs')
  .select('*, profiles(display_name, ticket_number)')
  .order('created_at', { ascending: false })
  .range(0, 99); // ページネーション必須
```

### 11.2 利用可能なビュー
- `daily_active_users`（date, active_users｜過去400日）
- `event_summary`（event_name, total_events, unique_users, first/last_occurrence｜直近30日）

### 11.3 連携・合意が必要な事項（2026-06-12）
1. **イベント名の共有**：§5 が現行の全リスト。ダッシュボードのフィルタ/ラベルを実態に合わせる。
2. **スタッフ手動付与の扱い**：ミニアプリは `activity_logs` に書かない。`stamp_history`(`manual_admin`/`MANUAL-ADJUST-*`) を「スタッフ操作」として表示するか、将来 `event_logs` に記録するか方針合意が必要（盲点#4）。
3. **手動付与は `stamp_history` を全削除してから再記録**（[manual/route.ts](../app/api/stamps/manual/route.ts)）→ 履歴表示・来院回数集計に影響する旨を共有。
4. **タイムスタンプのUTC/JST差異**：`MANUAL-ADJUST-…HHMMSS` のHHMMSSはサーバー(UTC)時刻。表示時はJST換算。
5. **保持期間400日・purge手動**の運用認識合わせ。

### 11.4 パフォーマンス
- 常にページネーション（`.range`）。
- 日付範囲（`.gte/.lte created_at`）を必ず指定。
- user_id / event_name / created_at にインデックスあり（§2.1）。
