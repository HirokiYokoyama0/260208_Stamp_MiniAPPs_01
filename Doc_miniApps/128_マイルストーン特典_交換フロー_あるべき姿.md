## 1. ステータス・ライフサイクル（正本）

| ステータス | 意味 | 患者アプリ表示 | 遷移元 |
|---|---|---|---|
| `available` | マイルストーン到達済み（未交換） | 「この特典と交換する」ボタン | （自動付与で新規）|
| `pending` | 患者が交換申請済み（受付待ち）| 「受付で確認中...」 | **available（更新）** |
| `completed` | 受付で提供完了 | 「交換完了」 | pending（または available）|
| `cancelled` | 無効化（スタンプ減少・誤操作）| 「キャンセル済み」 | available / pending |
| `expired` | 有効期限切れ | 「有効期限切れ」 | pending |

**重要な設計原則**：`pending` の遷移元は **`available`（＝既存レコードの更新）**。マイルストーン1つにつき `reward_exchanges` は**常に1行**であるべき（新規INSERTで増やさない）。

## 2. あるべき姿：患者（ユーザー）フロー

```
① 来院 → 受付QRを読む（カメラQR / アプリ内スキャン）
        → 来院スタンプ付与（優良15 / 通常10）
② マイルストーン到達（10・50・300…）
        → システムが特典を status='available' で自動付与（1行）
        → milestone_history に到達記録
③ 特典タブに「交換可能！」＋「この特典と交換する」表示
④ 患者が「この特典と交換する」を押す
        → 【あるべき】その available 行を status='pending' に UPDATE（行は増やさない）
        → 「受付で確認中...」表示・有効期限カウント開始
⑤ 受付で現物を受け取る
        → スタッフ操作で status='completed'
        → 「交換完了」表示
```

**補足**
- スタンプは積み上げ式で**減らない**（交換してもスタンプ数は不変）。
- 優先度ルール：同一マイルストーンで複数該当時は premium_menu > poic > toothbrush の最上位のみ。
- 有効期限：歯ブラシ＝当日23:59 / POIC・選べるメニュー＝5ヶ月。期限切れは `expired`。

## 3. あるべき姿：スタッフフロー（管理ダッシュボード）

```
A. 患者がアプリで交換済み（pending がある場合）
   → ダッシュボードで pending（「特典交換:…」）を確認
   → 現物を渡す → 「引き渡し完了」→ その行を status='completed' に UPDATE
   → 1行のまま completed（重複しない）

B. 患者がアプリで交換していない（available のまま来院した場合）
   → スタッフが available を直接「引き渡し完了」→ status='completed' に UPDATE
   → 1行のまま completed（重複しない）

C. スタンプ誤付与の是正等
   → 該当マイルストーン特典を status='cancelled'（論理削除）
```

**重要な設計原則**：スタッフの「引き渡し完了」は**対象1行を completed に更新するだけ**。同一マイルストーンに複数行を作らない／残さない。A・Bどちらでも**最終的に1行**になるのが正。

## 4. 状態遷移図（正本）

```
                 マイルストーン到達（自動付与・1行）
                              │
                              ▼
                        [ available ]
                       ／            ＼
   患者がアプリで交換 ／              ＼ スタッフが直接引き渡し(B)
       (UPDATE)     ▼                ▼   (UPDATE)
                [ pending ] ──引き渡し完了(A)──►[ completed ]
                    │        (UPDATE)
       期限切れ     │
        (UPDATE)   ▼
                [ expired ]

   スタンプ減少・誤操作 → [ cancelled ]（available/pending から・論理削除）
```

**不変条件**：`(user_id, reward_id, milestone_reached)` につき有効行は**常に1行**。

## 5. 現状（実装）との差分＝重複バグ

| 項目 | あるべき姿 | 現状の実装 | 影響 |
|---|---|---|---|
| 交換時（④） | 既存 `available` を `pending` に **UPDATE**（1行維持） | [`/api/rewards/exchange`](../app/api/rewards/exchange/route.ts#L210) が **新規INSERT**（`pending`チェックのみで`available`を見ない）| available が取り残され**2行＝重複** |
| スタッフ完了（A）| pending を completed に更新 | ダッシュボード `/api/reward-exchanges/{id}/complete` が指定id 1行を更新（**兄弟availableをdedupしない**）| 取り残しavailableが残存 |

- 重複の必要十分条件：**available事前存在 × exchange新規作成**（[127](127_マイルストーン特典重複_ミニアプリ開発者へ確認.md) §3-3、横山123459 ms30=重複/ms100=重複なしで実証）。
- 修正＝④を「update-not-insert（条件付きアトミックUPDATE）」に。これで**あるべき姿（1行維持）**に一致する。

## 6. 実装・移行時の絶対条件（患者の交換を壊さない）

- 順序厳守：**B（LIFF update-not-insert・fail-open）→ C（既存重複クリーンアップ・論理削除）→ D-fix（部分ユニークインデックス）**。逆順は患者の交換が失敗する。
- **D-fix は「フルUNIQUE」ではなく「部分ユニークインデックス」**（`WHERE status IN ('available','pending','completed')`）。論理削除(cancelled)と両立し、有効行を1マイルストーン1行に保証（05ドキュメントのフルUNIQUE記載は論理削除モデルでは破綻＝誤り）。
- LIFF修正は **fail-open**：available有→UPDATE / 無→INSERT / 想定外エラー時も従来INSERTにフォールバック（「今まで成功した交換が失敗する」経路を作らない）。
- 詳細は [127](127_マイルストーン特典重複_ミニアプリ開発者へ確認.md) §3-4。

### 6-1. 前提：`reward_exchanges` の anon UPDATE を許可するRLSポリシー（適用済み・2026-07-19）

LIFFのexchangeが `available→pending` を **UPDATE** するには、`reward_exchanges` に anon UPDATE を許可するRLSポリシーが必要（`003`はSELECT/INSERTのみ・LIFFはservice role不可＝[107](107_RLSポリシー修正_Service_Role_Key不要化.md)）。
- 適用：`supabase/033_allow_anon_update_reward_exchanges_available_to_pending.sql`（**anon: available→pending のみ許可**・最小権限）を本番Supabaseに手動適用（2026-07-19 14:50 UTC）。
- これが無いと update-not-insert は0行→INSERTにフォールバックし機能しない（ms40/60で確認）。適用後 ms70 で **1行更新（created旧・兄弟cancelなし）＝稼働確定**。
- ⚠️ 本ポリシーは available→pending のみ。`cancel`/`expired`（[106](106_バグ報告_スタンプ減少時の特典削除未対応.md)）は対象外で別タスク。
