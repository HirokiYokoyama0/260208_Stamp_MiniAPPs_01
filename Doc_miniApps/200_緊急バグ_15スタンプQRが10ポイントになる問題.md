**実際の動作**:
```

  ↓

  ↓
10ポイント付与
  ↓
イベントログ: type="regular", stamps_added=10
```

### 問題B: 購買インセンティブが1回のみ

## 💡 原因の仮説

### 仮説1: QRコード生成の問題（可能性: 高）

**内容**: 管理ダッシュボード側でQRコードを生成する際、間違ったペイロードを生成している

**検証方法**:
1. 管理ダッシュボードのQRコード生成機能を確認
2. 15スタンプQRのペイロードが正しいか確認
   - カメラQR: `?userId=...&amount=15&type=qr&location=premium`
   - ミニアプリQR: `{"type":"premium","stamps":15}`

**根拠**:
- イベントログに `regular` として記録されている
- これはAPIが受け取ったペイロードが `regular` だったことを意味する

**対策**:
- 管理ダッシュボード開発者に連絡
- QRコード生成ロジックを確認
- 既存のQRコードを再生成

---

### 仮説2: QRコードの混在（可能性: 高）

**内容**: 院内に古いQRコード（10スタンプ）と新しいQRコード（15スタンプ）が混在している

**検証方法**:
1. 院内の全QRコードをスキャンして確認
2. 各QRコードのペイロードを記録
3. 混在していないか確認

**根拠**:
- 「同じiPhoneでもうまくいく時とうまくいかない時がある」
- これは「異なるQRコードをスキャンしている」可能性を示唆

**対策**:
- 古いQRコードを廃棄
- 新しいQRコードのみを使用

---

### 仮説3: purchase_incentive判定の誤り（可能性: 中）

**内容**: 購買インセンティブが誤って1日1回制限の対象になっている

**現在の実装**:

#### カメラQRスキャン (`/api/stamps/auto`)
```typescript
// 47-49行目
if (location !== "shop") {
  // 1日1回制限チェック
}
```
- `location === "shop"` なら制限をスキップ ✅

#### ミニアプリ内QRスキャン (`/api/stamps/scan`)
```typescript
// 110行目
if (type !== "purchase") {
  // 1日1回制限チェック
}
```
- `type === "purchase"` なら制限をスキップ ✅

**問題の可能性**:
- QRコード生成時に `location="shop"` または `type="purchase"` が正しく設定されていない
- 購買インセンティブQRが `location="premium"` や `type="regular"` として生成されている

**検証方法**:
1. イベントログで購買インセンティブのログを確認
2. `type` フィールドが `"purchase"` か `"regular"` か確認
3. `"regular"` なら、QRコード生成が間違っている

**対策**:
- QRコード生成時のペイロードを修正

---

### 仮説4: スタンプ個数の取り違え（可能性: 低）

**内容**: QRコード生成時に `amount` や `stamps` の値を取り違えている

**検証方法**:
- イベントログで `stamps_added` を確認
- 15スタンプQRで `stamps_added: 10` になっていないか

**対策**:
- QRコード生成ロジックを修正

---

### 仮説5: LIFF scanCodeV2のバグ（可能性: 極低）

**内容**: LIFF SDKの `scanCodeV2()` が誤ったQRコードを返している

**検証方法**:
- イベントログの `qr_raw_value` を確認（現在は記録されていない）
- クライアントサイドのログを確認

**根拠**:
- 「別の携帯では15ポイント問題なく付与された」
- これはLIFFの問題ではない可能性が高い

## 🔬 調査に必要なデータ

### 1. イベントログの確認

Supabase SQL Editorで以下のクエリを実行:

```sql
-- 最近のスタンプスキャンログを確認
SELECT
  created_at,
  event_name,
  user_id,
  metadata->>'type' AS stamp_type,
  metadata->>'stamps_added' AS stamps_added,
  metadata->>'scan_method' AS scan_method,
  metadata->>'qr_raw_value' AS qr_raw_value,
  metadata->>'qr_parsed' AS qr_parsed
FROM event_logs
WHERE event_name IN ('stamp_scan_success', 'stamp_scan_success_in_app', 'stamp_scan_fail')
  AND created_at >= CURRENT_DATE - INTERVAL '7 days'
ORDER BY created_at DESC
LIMIT 50;
```

**確認ポイント**:
- `stamp_type` が `"regular"` になっている15スタンプQRスキャンがないか
- `stamps_added` が `10` になっている優良患者スキャンがないか
- `scan_method` が `"camera"` か `"in_app"` か

---

### 2. stamp_historyの確認

```sql
-- 最近のスタンプ履歴を確認
SELECT
  visit_date,
  user_id,
  stamp_method,
  amount,
  stamp_number,
  notes
FROM stamp_history
WHERE visit_date >= CURRENT_DATE - INTERVAL '7 days'
  AND stamp_method IN ('qr', 'purchase_incentive')
ORDER BY visit_date DESC
LIMIT 100;
```

**確認ポイント**:
- `amount=10` で `notes` に「premium」が含まれているレコードがないか
- `stamp_method='qr'` で `notes` に「購買」が含まれているレコードがないか

## ✅ 購買インセンティブ問題の解決

**修正完了日**: 2026-03-28

### 問題

購買インセンティブQR（5スタンプ）が1日1回しかスキャンできない問題が発生していました。

### 根本原因

**[app/api/stamps/route.ts](../app/api/stamps/route.ts)** の重複チェック（Lines 46-78）が、QRペイロードのタイプを確認せずに**全てのQRコードに1日1回制限を適用していた**ことが原因でした。

### 修正内容

1. QRペイロードをパースして `type` を取得
2. `type === "purchase"` の場合は重複チェックをスキップ
3. `stamp_method` を正しく設定（`purchase_incentive` vs `qr`）

**詳細**: [Doc 105: 緊急バグ修正_購買インセンティブ1日1回制約問題](105_緊急バグ修正_購買インセンティブ1日1回制約問題.md)

### 修正結果

✅ 購買インセンティブQRは1日何度でもスキャン可能
✅ 通常QR・優良QRは1日1回のみ（従来通り）
✅ ビルドエラーなし
