# ダッシュボード側への確認事項：016Bマイグレーション適用について

## 作成日
2026-03-15

## 目的
LIFFアプリ側で「本日のQRスキャンを削除」機能が正常に動作しない問題について、データベースマイグレーション（016B）の適用可否を確認する。

---

## 🚨 現在の問題

### 症状
- LIFFアプリの「本日のQRスキャンを削除」ボタンを押すと「削除成功」と表示される
- しかし、実際にはデータベースから削除されていない（削除件数: 0件）
- Vercelログには「1件削除」と表示されるが、データベースには残っている

### エビデンス

#### Vercelログ（2026-03-15）
```
POST 200 /api/stamps/scan/delete-today
[Delete Today QR] 削除成功: 1件削除, -10ポイント
```

#### データベース実態
```
ID: 73741219-fdd6-4475-99a5-66cb8ed7cc0b
user_id: U5c70cd61f4fe89a65381cd7becee8de3
stamp_number: 30
amount: 10
visit_date: 2026-03-15T01:36:30.677Z
stamp_method: qr_scan
```
→ **削除されていない**

#### テストスクリプト結果
```bash
npx tsx test/test-delete-today-qr.ts

✅ 削除成功: 0件削除
削除前 stamp_count: 30
削除後 stamp_count: 30
差分: 0 (期待値: -10)
```

## 🔍 原因の仮説

### 仮説1: DELETEトリガーが適用されていない
- ダッシュボード側では `trigger_update_profile_on_stamp_delete` が適用済みと認識
- しかし、LIFFアプリ側（ANON_KEY使用）では実際に削除できていない
- **可能性**: RLSポリシーまたはトリガーが本番環境に適用されていない

### 仮説2: 権限の違い
- ダッシュボード側: `staff01` アカウント（管理者権限？）で削除可能
- LIFFアプリ側: `ANON_KEY` で削除→権限不足で実際には削除されない
- **可能性**: RLS DELETE ポリシーが設定されていない

### 仮説3: クライアント設定の違い
- ダッシュボード側: `SERVICE_ROLE_KEY` 使用（RLSバイパス）
- LIFFアプリ側: `ANON_KEY` 使用（RLS適用）
- **可能性**: ANON_KEYでは DELETE が許可されていない

---

## 📋 提案する解決策：016Bマイグレーション適用

### マイグレーションファイル
`supabase/016B_add_delete_policy_stamp_history.sql`

### 実行内容

#### 1. DELETE ポリシーの追加
```sql
CREATE POLICY "allow_public_delete"
  ON stamp_history
  FOR DELETE
  USING (true);
```
→ ANON_KEY でも削除可能にする

#### 2. UPDATE ポリシーの追加
```sql
CREATE POLICY "allow_public_update"
  ON stamp_history
  FOR UPDATE
  USING (true);
```
→ ANON_KEY でも更新可能にする

#### 3. DELETE トリガーの追加
```sql
CREATE OR REPLACE FUNCTION update_profile_on_stamp_delete()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE profiles
  SET
    stamp_count = (
      SELECT COALESCE(MAX(stamp_number), 0)
      FROM stamp_history
      WHERE user_id = OLD.user_id
    ),
    visit_count = (
      SELECT COUNT(*)
      FROM stamp_history
      WHERE user_id = OLD.user_id AND amount = 10
    ),
    last_visit_date = (
      SELECT MAX(visit_date)
      FROM stamp_history
      WHERE user_id = OLD.user_id
    ),
    updated_at = NOW()
  WHERE id = OLD.user_id;
  RETURN OLD;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_update_profile_on_stamp_delete
AFTER DELETE ON stamp_history
FOR EACH ROW
EXECUTE FUNCTION update_profile_on_stamp_delete();
```
