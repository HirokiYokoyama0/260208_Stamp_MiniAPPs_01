# デバッグ: 削除後の再スキャン問題調査

**調査日**: 2026-03-28
**報告**: ユーザーが削除後に再スキャンできない問題

---

## 🐛 問題の症状

1. カメラQRスキャンまたはミニアプリ内QRスキャンでスタンプ付与
2. スタッフ機能で本日のQRスキャンを削除
3. **削除後に再度「来院スタンプを読み取る」ボタンを押す**
4. ❌ **「本日は既にQRコードでスタンプを受け取っています」とエラー表示**

→ **削除したのに、まだ「今日スキャン済み」と判定されている**

---

## 🔍 調査項目

### 1. Supabase実データの確認

以下のSQLクエリをSupabase SQL Editorで実行して、実データを確認してください。

#### クエリ1: 本日のQRスタンプ履歴を確認

```sql
-- 対象ユーザーID（実際のLINEユーザーIDに置き換えてください）
DO $$
DECLARE
  target_user_id TEXT := 'Uxxxxxxxxxxxxxxx';  -- ★ 実際のユーザーIDに変更
  today_jst DATE;
  start_of_day TIMESTAMPTZ;
  end_of_day TIMESTAMPTZ;
BEGIN
  -- 日本時間（JST）で今日の日付を計算
  today_jst := (CURRENT_TIMESTAMP AT TIME ZONE 'Asia/Tokyo')::DATE;
  start_of_day := (today_jst || ' 00:00:00')::TIMESTAMP AT TIME ZONE 'Asia/Tokyo';
  end_of_day := (today_jst || ' 23:59:59.999')::TIMESTAMP AT TIME ZONE 'Asia/Tokyo';

  RAISE NOTICE '==========================================';
  RAISE NOTICE '対象ユーザーID: %', target_user_id;
  RAISE NOTICE '今日の日付（JST）: %', today_jst;
  RAISE NOTICE '範囲: % ~ %', start_of_day, end_of_day;
  RAISE NOTICE '==========================================';
END $$;

-- 本日のQRスタンプ履歴を表示
SELECT
  id,
  user_id,
  visit_date,
  stamp_method,
  amount,
  stamp_number,
  notes,
  created_at
FROM stamp_history
WHERE user_id = 'Uxxxxxxxxxxxxxxx'  -- ★ 実際のユーザーIDに変更
  AND stamp_method = 'qr'
  AND visit_date >= (CURRENT_DATE AT TIME ZONE 'Asia/Tokyo')
  AND visit_date < (CURRENT_DATE AT TIME ZONE 'Asia/Tokyo' + INTERVAL '1 day')
ORDER BY visit_date DESC;
```

**期待される結果**:
- 削除前: 1件以上のレコードが表示される
- 削除後: **0件** （レコードが完全に削除されている）

### 3. 仮説と検証

#### 仮説1: タイムゾーン計算の不一致

**内容**: 削除APIとスキャンAPIでタイムゾーン計算が異なる

**削除API** (`delete-today/route.ts` 58-61行目):
```typescript
const nowJST = new Date(new Date().getTime() + 9 * 60 * 60 * 1000);
const todayJST = nowJST.toISOString().split("T")[0];
const startOfDay = `${todayJST}T00:00:00+09:00`;
const endOfDay = `${todayJST}T23:59:59.999+09:00`;
```

**スキャンAPI** (`scan/route.ts` 112-115行目):
```typescript
const nowJST = new Date(new Date().getTime() + 9 * 60 * 60 * 1000);
const todayJST = nowJST.toISOString().split("T")[0];
const startOfDayUTC = `${todayJST}T00:00:00+09:00`;
const endOfDayUTC = `${todayJST}T23:59:59.999+09:00`;
```

**結論**: **完全に同一のロジック** → 仮説1は否定

---

#### 仮説2: Supabaseキャッシュ

**内容**: Supabaseクライアントがクエリ結果をキャッシュしている

**検証方法**:
1. 削除API実行後、数秒待ってから再スキャン
2. それでも問題が起きるなら、キャッシュではない

---

#### 仮説3: RLSポリシーの問題

**内容**: RLSポリシーがDELETEを許可しているが、実際には削除されていない

**検証方法**:
- Supabase SQL Editorで直接確認（クエリ1）
- `stamp_history` テーブルのDELETEポリシーを確認

```sql
-- RLSポリシーを確認
SELECT
  schemaname,
  tablename,
  policyname,
  permissive,
  roles,
  cmd,
  qual,
  with_check
FROM pg_policies
WHERE tablename = 'stamp_history';
```

**期待される結果**:
```
policyname: allow_public_delete
cmd: DELETE
permissive: PERMISSIVE
roles: {public}
qual: true
```

---

#### 仮説4: トリガーが実行されていない

**内容**: DELETEトリガーがエラーで失敗し、トランザクションがロールバックされている

**検証方法**:
- Vercelログで「削除クエリ結果」を確認
- `count` が削除対象の件数と一致するか
- `error` が `null` か

---

#### 仮説5: フロントエンドのキャッシュ

**内容**: フロントエンドが古いデータをキャッシュしている

**検証方法**:
1. 削除後、ページをリロード
2. それでも問題が起きるなら、フロントエンドキャッシュではない

## 🚨 緊急対応が必要な場合

もしSupabase実データで「削除後もレコードが残っている」場合:

### 対策1: DELETEトリガーの確認

```sql
-- トリガーが存在するか確認
SELECT
  trigger_name,
  event_manipulation,
  event_object_table,
  action_statement
FROM information_schema.triggers
WHERE event_object_table = 'stamp_history'
  AND trigger_name LIKE '%delete%';
```

### 対策2: 手動削除

```sql
-- 対象ユーザーの本日QRスタンプを手動削除
DELETE FROM stamp_history
WHERE user_id = 'Uxxxxxxxxxxxxxxx'  -- ★ 実際のユーザーIDに変更
  AND stamp_method = 'qr'
  AND visit_date >= (CURRENT_TIMESTAMP AT TIME ZONE 'Asia/Tokyo')::DATE
  AND visit_date < ((CURRENT_TIMESTAMP AT TIME ZONE 'Asia/Tokyo')::DATE + INTERVAL '1 day');
```
