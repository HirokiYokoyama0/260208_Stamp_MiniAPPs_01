## 0. 要旨

LIFFアプリに **3つの新しいイベント名** を追加し、本番（`main`）へデプロイ済みです。
すべて既存 `event_logs` テーブルへの追記で、**スキーマ変更・RLS変更はありません**。
ダッシュボードの**イベント名フィルタ／ラベル定義への追加**をお願いします。

| 新 event_name | 発生源 | 目的 |
|---------------|--------|------|
| `auto_stamp_entry` | LIFF（クライアント / 会員証ホーム） | カメラQRで開かれた時の「着地」を記録（何のQR/転送有無） |
| `auto_stamp_result` | LIFF（クライアント / auto-stampページ） | カメラQR付与処理の「結末」を記録（成功/失敗/例外） |
| `staff_manual_stamp` | LIFF（サーバーAPI / 手動付与） | スタッフ手動付与の事実を記録 |

加えて、**全クライアントイベントの `metadata` に `page_query`（URLクエリ文字列）が追加**されました。

> 背景：「カメラQRを読んだのにスタンプが付かない」事象（例：会員証8305 宇野様）の原因が、従来ログでは追えなかったため（クライアント側処理が無記録だった）。本計装で再発時に原因特定が可能になります。

---

## 1. 共通事項

- 保存先：既存 `event_logs`（`id, user_id, event_name, source, metadata(jsonb), created_at`）。**新テーブルなし**。
- 閲覧：従来どおり **SERVICE_ROLE_KEY**（RLSバイパス）。anonからのSELECTは不可（変更なし）。
- 既存ビュー `event_summary` には新イベントが**自動的に集計対象**として現れます。
- クライアントイベントには従来どおり自動付与メタ（`user_agent`/`screen_size`/`viewport_size`/`referrer`/`page_path`/**`page_query`(新)**/`timestamp`）が入ります。
- **サーバー発の `staff_manual_stamp` には自動付与メタは付きません**（`window`が無いため）。

---

## 2. イベント別 metadata 仕様

### 2.1 `auto_stamp_entry`（クライアント／会員証ホーム `app/page.tsx`）

カメラQR等で**URLパラメータ付きでホームが開かれた時だけ**発火（通常のホーム閲覧では発火しない）。

| metadata キー | 型 | 説明 |
|---------------|----|------|
| `raw_query` | string | `window.location.search`（例: `?action=stamp&type=qr&amount=15&location=premium`） |
| `action` | string\|null | URLの`action`（通常 `stamp`） |
| `type` | string\|null | `qr` / `purchase` |
| `amount` | string\|null | スタンプ数の文字列 |
| `location` | string\|null | `premium` 等 |
| `redirected` | boolean | `/auto-stamp` へ転送したか（= **その時点のURL**が有効なスタンプQRだったか） |
| `is_in_client` | boolean\|undefined | 予約フィールド（現状未設定のことあり） |
| `liff_state` | string\|null | 第1段階の `liff.state`（デコード済み）。例: `?action=stamp&type=qr&amount=15&location=entrance` |
| `from_liff_state` | boolean | `action/type/amount` を `liff.state` から補完したか（true=第1段階の着地） |

> ⚠️ **`user_id` は基本的に `null`**。理由：ホーム画面で `liff.init` の二重呼び出し（iOS無限ループの再発）を避けるため、ここでは LINE プロフィールを取得していません。`created_at` と `app_open` 等で時系列突合してください。

#### カメラQRは「2段階」で着地する（重要）

LINEからカメラQRでLIFFを開くと、**`auto_stamp_entry` が通常2回**発火します。これは重複ではなく正規の挙動です。

| 段階 | raw_query | action/type/amount | redirected | from_liff_state |
|------|-----------|--------------------|-----------|-----------------|
| 第1段階（着地直後） | `?liff.state=%3Faction%3D...`（本来のクエリがエンコードされて埋まっている） | URL上は読めない → **`liff_state`をデコードして補完** | **false**（まだ`/auto-stamp`へ進めない） | **true** |
| 第2段階（展開後） | `?action=stamp&type=qr&amount=15&...`（LIFF SDKが展開） | そのまま読める | **true**（`/auto-stamp`へ転送） | false |

**ダッシュボードでの解釈指針**：
- `from_liff_state=true`（=第1段階）は `redirected=false` が正常。**それ単体では失敗ではありません**。
- 「カメラQRを読んだのに付与されない」失敗は、**第1段階の `auto_stamp_entry` はあるのに、直後に第2段階（`redirected=true`）も `auto_stamp_result` も続かない**パターンで特定できます（例: 会員証8305 宇野様の症状＝`app_open`のみでスキャン系イベント無し）。
- 第1段階でも `liff_state` / 補完後の `type`・`amount` から「何のスタンプQRを読もうとしたか」が分かります。

### 2.2 `auto_stamp_result`（クライアント／`app/auto-stamp/page.tsx`）

カメラQR付与処理の結末。`outcome` で分岐を区別。

| metadata キー | 型 | 説明 |
|---------------|----|------|
| `outcome` | string | 下表参照 |
| `raw_query` | string | URLクエリ文字列 |
| `parsed_amount` | number\|undefined | パース後スタンプ数 |
| `is_in_client` | boolean\|undefined | LINEアプリ内か |
| `liff_version` | string\|undefined | LIFF SDKバージョン |
| `http_status` | number\|undefined | `/api/stamps/auto` のHTTPステータス（API到達時） |
| `error_message` | string\|undefined | 失敗・例外時のメッセージ |

**`outcome` の値**:

| outcome | 意味 | user_id |
|---------|------|---------|
| `not_in_client` | LINEアプリ外で開かれた | なし |
| `not_logged_in` | 未ログイン（liff.loginへ） | なし |
| `invalid_params` | action/type不正 | なし |
| `invalid_amount` | スタンプ数が5/10/15以外 | なし |
| `api_success` | 付与成功 | **あり** |
| `api_fail` | API失敗（重複/エラー等。`error_message`に詳細、`alreadyReceived`含む） | **あり** |
| `exception` | 処理中の例外（liff.init失敗等） | なし |

> 「カメラQRを読んだのに付与されない」は、**`auto_stamp_entry`（redirected=true）はあるのに `auto_stamp_result` の `api_success` が無い**、というパターンで特定できます。

### 2.3 `staff_manual_stamp`（サーバー／`app/api/stamps/manual`）

スタッフがLIFFミニアプリ画面で手動付与/調整した時に発火（付与成功後）。

| metadata キー | 型 | 説明 |
|---------------|----|------|
| `before` | number | 変更前スタンプ数 |
| `after` | number | 変更後スタンプ数 |
| `change_amount` | number | 差分（after - before） |
| `deleted_history_count` | number\|undefined | 削除した履歴件数（現状未設定） |
| `staff_initiated` | boolean | 常に `true` |
| `staff_id` | null | **現状 null**（下記「未解決事項」参照） |

- `user_id` = **対象患者**のLINE User ID。
- サーバー発のため `user_agent` 等の自動メタは付きません。

> ⚠️ **`staff_id` は現状 `null`**。これは文書121 確認事項④への回答そのものです。LIFF側のスタッフ操作は**共有PIN（`NEXT_PUBLIC_STAFF_PIN`）認証のみ**で、個々のスタッフを識別する情報を持っていません。「誰が」を残すにはミニアプリへのスタッフ選択/ログイン機構の追加が必要で、別途設計が必要です。
> なお**ダッシュボード発の手動変更は従来どおり `activity_logs.staff_id` で識別可能**です（こちらはミニアプリの本イベントとは別系統）。

## 3. ダッシュボード側への依頼

1. **イベント名の登録**：フィルタ・ラベル・凡例に下記3つを追加
   - `auto_stamp_entry`（カメラQR着地）
   - `auto_stamp_result`（カメラQR結末）
   - `staff_manual_stamp`（スタッフ手動付与）
2. （任意）**スタッフ手動付与の可視化**：`staff_manual_stamp` を「手動付与」一覧として表示可。ただし `staff_id=null` のため**LIFF発は担当者特定不可**。担当者特定が必要な手動変更は、ダッシュボード発（`activity_logs`）側に寄せる運用を推奨。
3. （任意）**スキャン不具合モニタ**：`auto_stamp_result` の `outcome != 'api_success'` を集計すると、カメラQR付与の失敗率・失敗理由が可視化できます。

## 4. 確認用SQL

```sql
-- 新イベントの最近の発生状況
select created_at, event_name,
       metadata->>'raw_query'     as raw_query,
       metadata->>'redirected'    as redirected,
       metadata->>'outcome'       as outcome,
       metadata->>'http_status'   as http_status,
       metadata->>'error_message' as error_message,
       user_id
from event_logs
where event_name in ('auto_stamp_entry','auto_stamp_result','staff_manual_stamp')
  and created_at >= current_date - interval '7 days'
order by created_at desc;

-- カメラQRの失敗内訳（outcome別）
select metadata->>'outcome' as outcome, count(*)
from event_logs
where event_name = 'auto_stamp_result'
  and created_at >= current_date - interval '30 days'
group by 1 order by 2 desc;

-- 「着地はしたが成功していない」候補（時系列突合のヒント）
-- auto_stamp_entry(redirected=true) が直近にあるのに api_success が無いケースを目視確認
select created_at, event_name, metadata->>'raw_query' q, metadata->>'outcome' outcome
from event_logs
where event_name in ('auto_stamp_entry','auto_stamp_result')
  and created_at >= current_date - interval '1 day'
order by created_at;

-- スタッフ手動付与の一覧
select created_at, user_id,
       metadata->>'before' as before, metadata->>'after' as after,
       metadata->>'change_amount' as change_amount
from event_logs
where event_name = 'staff_manual_stamp'
  and created_at >= current_date - interval '30 days'
order by created_at desc;
```

## 6. 未解決事項・別件（共有）

- **`staff_id` 付与**：LIFF側にスタッフ識別子なし（共有PIN運用）。担当者記録が必要ならミニアプリにスタッフ選択/ログイン追加を別途設計。
- **【別件・保留中】来院回数 `visit_count` の数え方**：実DBの稼働トリガーが `amount = 10` 基準で、修正版 `021`（`stamp_method` 基準）が未適用であることを確認済み。これにより「優良15来院の過少カウント」「手動±10の誤カウント」が発生し得ます。**本連携（ログ追加）とは別件**で、対応方針は保留中。詳細は別途。
