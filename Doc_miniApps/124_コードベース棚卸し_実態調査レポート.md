### 1.3 🔴 重複・整合性の問題：スタンプ付与が3系統

スタンプを付与するAPIが**3つ独立に存在**し、それぞれが「重複チェック・マイルストーン付与・イベントログ」を**個別実装**している。

| ルート | 入力方式 | 1日1回制限 | 5秒重複防止 | イベントログ | 備考 |
|--------|---------|-----------|------------|------------|------|
| `/api/stamps` | `qrCodeId`（QR生値） | あり | （要確認） | success/fail | **会員証画面のQRScanner**用・旧来 |
| `/api/stamps/scan` | `{type, stamps, qrCodeId}` | あり | あり | success_in_app/fail | /scanページ用 |
| `/api/stamps/auto` | `{amount, type, location}` | あり | あり | success(camera)/fail | カメラQR用 |

**問題点**:
- 入力フォーマットがバラバラ（qrCodeId / payload / action param）で、同じ「来院スタンプ」を3経路で付与している。
- ロジック（重複判定の方法・JST境界・マイルストーン呼び出し）が各ファイルにコピーされており、**修正漏れ・仕様差の温床**（過去の「15ポイント問題」「二重付与」もこの分散が一因）。
- さらに**アプリ内スキャンUIが2つ**ある：会員証画面の `QRScanner`(→`/api/stamps`) と 専用 `/scan`ページ(→`/api/stamps/scan`)。現状アプリ内スキャンは一時無効化（[123](123_アプリ内スキャン機能_一時無効化.md)）。

> 改善方向（提案・本レポートでは変更しない）：付与ロジックを1つの内部関数に集約し、3ルートは入力アダプタとして薄くする。ただし運用中のため段階的に。

## 7. 代表ケース：初回ユーザーのカメラQR直読み → `/api/stamps/auto` 500（2026-07-15追記）

### 7.1 事象
会員証**2976 森田千代乃**さん：カメラで来院QRを読んだがスタンプが付与されず、スタッフが手動付与で対応。Vercelログ（サーバー）とevent_logs（クライアント）の突合で原因確定。

### 7.2 根本原因
**プロフィール(`profiles`)のUPSERTは会員証/ホームタブ（[AdultHome.tsx:93-109](../components/(adult)/AdultHome.tsx#L93)）到達時のみ**実行される。カメラQRの入口 `/auto-stamp` はここを通らない。
→ **ホーム未経由の初回ユーザーがカメラQRを読むと、`profiles`に行が無いまま [`/api/stamps/auto`](../app/api/stamps/auto/route.ts#L152) が `.single()` を実行し `PGRST116（0 rows）→ 500`**。付与されない。LINEログイン許諾自体は成立（userId取得済み）＝認証エラーではない。

### 7.3 時系列（JST・2026/07/15）
| 時刻 | 記録元 | 出来事 |
|------|--------|--------|
| 13:01:12 | event_logs / **Vercel** | カメラQR→`/auto-stamp`→`/api/stamps/auto` → **500（profiles 0 rows）** |
| 13:01:23 | Vercel | `/api/users/me` → 404（同一原因）|
| 13:02:14 | event_logs / **Vercel** | 再スキャン → また **500** |
| **13:02:29** | profiles | `profiles.created_at`（初めてホーム描画で登録）|
| 13:11:02 | event_logs | `app_open`（クエリ無し＝通常起動）|
| 13:12:54 | stamp_history | **スタッフ手動付与 +15（0→15）** |
| 13:13:46 / 13:17:17 | event_logs | スタンプページ閲覧（15確認）|

### 7.4 補足
- 発生頻度は低い（多くの患者は正常完走）。Vercel期間内(07/15昼〜夜)の`/api/stamps/auto` 500は全て本原因（PGRST116）。
- iOS別のクライアント失敗（18.7=`not_in_client` / 26.5=応答遅延6-9s）とは**別系統**。同時アクセスは主因ではない（concurrency max=3）。

### 7.5 対処方針（案・基本フロー非破壊が絶対条件）
> 現状ほとんど正常動作しているため、**ハッピーパス（profile存在時）には一切手を入れず**、「profile未存在の時だけ」フォールバックを足す方針。

**推奨A（サーバー側・最小・happy path非干渉）**：[`/api/stamps/auto`](../app/api/stamps/auto/route.ts#L158) の `if (profileError || !profile)` 分岐で、`profileError.code === 'PGRST116'`（0行）の時だけ**最小プロフィールをUPSERTしてから付与を継続**（`profile = { stamp_count: 0, family_id: null }`）。表示名は後日ホーム訪問時のUPSERTで補完。それ以外のエラーは従来どおり500。
- happy path（profile存在）は`.single()`が成功 → **完全に無変更**。
- amount/visit_count/付与ロジック/RLS/トリガーには**非干渉**（profile未存在時のハンドリング追加のみ）。
- 任意拡張：`/auto-stamp`クライアントが既に持つ `liff.getProfile()` の `displayName/pictureUrl` をAPIに渡せば、作成プロフィールに名前も入る。
- 注意：`profiles`へのINSERTがRLS(026B・id形式チェック)で anon 許可されるか要確認。不可なら当該UPSERTのみ `getSupabaseAdmin()` 使用。

**代替D（付与しない・案内のみ）**：profile未存在時は500をやめ「アプリを一度開いてから再度お試しください」を返す。実装最小・ゼロリスクだが**その場では付与されない**（UX劣る）。

**非推奨**：付与3系統すべてに同ガードを入れる／UPSERTをAppLayoutへ移設（影響範囲が広く、基本フローを壊すリスク）。

→ **結論：推奨A（サーバー側フォールバック）が「基本フロー非破壊」と「初回でも付与」を両立**。実装は承認の上で段階リリース＋ロールバック前提。
