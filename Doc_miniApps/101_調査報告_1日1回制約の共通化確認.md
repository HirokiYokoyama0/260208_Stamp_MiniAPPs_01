## 📋 調査結果サマリー

### 1. カメラQRスキャン (`/api/stamps/auto`)

**ファイル**: [app/api/stamps/auto/route.ts](../app/api/stamps/auto/route.ts)
**該当行**: 47-99

#### 1日1回制約チェックのコード

```typescript
// 47-49行目: 購買インセンティブは制約対象外
if (location !== "shop") {
  // 日本時間（JST = UTC+9）で当日の範囲を計算
  const nowJST = new Date(new Date().getTime() + 9 * 60 * 60 * 1000);
  const todayJST = nowJST.toISOString().split("T")[0];
  const startOfDayUTC = `${todayJST}T00:00:00+09:00`;
  const endOfDayUTC = `${todayJST}T23:59:59.999+09:00`;

  // 56-62行目: stamp_historyから当日のQRスタンプを検索
  const { data: todayQrRecords, error: qrCheckError } = await supabase
    .from("stamp_history")
    .select("id, stamp_method, notes")
    .eq("user_id", userId)
    .eq("stamp_method", "qr")  // ★ 重要: QRスタンプのみチェック
    .gte("visit_date", startOfDayUTC)
    .lt("visit_date", endOfDayUTC);

  // 72-99行目: 既に取得済みなら409 Conflictを返す
  if (todayQrRecords && todayQrRecords.length > 0) {
    console.log(`⚠️ 1日1回制限: User ${userId} は本日既にQRスタンプを取得済み`);
    return NextResponse.json(
      {
        success: false,
        message: "本日は既にQRコードでスタンプを受け取っています",
        alreadyReceived: true
      },
      { status: 409 }
    );
  }
}
```

## 🎯 重要なポイント

### 1. 共通化の鍵: `stamp_method = 'qr'`

両方のAPIが **同じ値 (`"qr"`)** を使って記録・チェックしているため、制約が共有されます。

```typescript
// カメラQR
.eq("stamp_method", "qr")  // チェック時
stamp_method: location === "shop" ? "purchase_incentive" : "qr"  // 記録時

// ミニアプリ内QR
.eq("stamp_method", "qr")  // チェック時
stamp_method: type === "purchase" ? "purchase_incentive" : "qr"  // 記録時
```

### 2. タイムゾーン処理の一貫性

両方のAPIで **完全に同じタイムゾーン計算** をしています：

```typescript
const nowJST = new Date(new Date().getTime() + 9 * 60 * 60 * 1000);
const todayJST = nowJST.toISOString().split("T")[0];
const startOfDayUTC = `${todayJST}T00:00:00+09:00`;
const endOfDayUTC = `${todayJST}T23:59:59.999+09:00`;
```

これにより、日本時間の「1日」が正確に共有されます。

### 3. 購買インセンティブの除外

両方のAPIで **同じ除外ロジック** を実装：

- カメラQR: `if (location !== "shop")`
- ミニアプリ内QR: `if (type !== "purchase")`

どちらも購買インセンティブを制約から除外しています。

## 📈 データベース設計の健全性

### `stamp_history` テーブルのレコード例

| id | user_id | visit_date | stamp_method | amount | notes |
|----|---------|------------|--------------|--------|-------|
| 1 | U123... | 2026-03-28T09:00 | qr | 10 | カメラ用QR (standard) |
| 2 | U123... | 2026-03-28T14:00 | qr | 15 | アプリ内スキャン (premium) |
| 3 | U123... | 2026-03-28T16:00 | purchase_incentive | 5 | 購買インセンティブ |

**重要**:
- レコード1と2は両方とも `stamp_method = 'qr'` なので、**互いに検出し合う**
- レコード3は `stamp_method = 'purchase_incentive'` なので、**制約チェックから除外**

しかし、**実際にはレコード2は記録されません**。なぜなら、レコード1が既に存在するため、14:00のスキャンは409 Conflictエラーになるからです。

## ✅ 結論

### 調査結果

**1日1回制約は正しく共通化されており、不整合はありません。**

### 共通化が保証される理由

1. **同じテーブル**: 両API共に `stamp_history` テーブルを使用
2. **同じカラム**: 両API共に `stamp_method = 'qr'` をチェック
3. **同じ時刻範囲**: 両API共にJST当日 00:00-23:59
4. **同じ除外条件**: 両API共に購買インセンティブを除外
5. **同じ記録方式**: 両API共に `stamp_method` に同じ値を記録

### 動作保証

- ✅ カメラQRスキャン後、ミニアプリ内スキャンは **ブロックされる**
- ✅ ミニアプリ内スキャン後、カメラQRスキャンは **ブロックされる**
- ✅ 購買インセンティブは **何度でもスキャン可能**
- ✅ 翌日になれば **再度スキャン可能**
